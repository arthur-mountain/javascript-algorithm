# Radix Sort（基數排序，LSD）

先按個位數排序，再按十位數排序……像郵局按區號分信。底層用 [Counting Sort](../counting-sort/README.md) 對每一位排序。Non-comparison-based。

## 觸發條件

整數排序，位數 d 有限且 `d × (n + k) < n log n`（即 d 較小，如手機號碼、身分證號、IP 位址）。

## 運作流程（執行期）

1. 找到最大值，確定最大位數 d。
2. 從最低位（LSD）到最高位，對每一位做一次**穩定排序**（此處用 Counting Sort，基數 k = 10）。
3. d 輪後排序完成。

## 逐步範例

`input = [170, 45, 75, 90, 2, 802, 24, 66]`（max = 802 → 3 位數 → 3 趟）

**第 1 趟（個位 exp=1）**：依個位數做穩定排序

```
170, 90, 802, 2, 24, 45, 75, 66
個位: 0    0   2   2   4   5   5   6
```

注意 170 在 90 前（個位都是 0，保留原序）、802 在 2 前（個位都是 2）→ 穩定。

**第 2 趟（十位 exp=10）**：在上一趟結果上，依十位數做穩定排序

```
802, 2, 24, 45, 66, 170, 75, 90
十位:  0   0   2   4   6   7    7   9
```

802 與 2 的十位都是 0，沿用上一趟「802 在 2 前」的順序；170 與 75 的十位都是 7，沿用上一趟「170 在 75 前」。

**第 3 趟（百位 exp=100）**：依百位數做穩定排序

```
2, 24, 45, 66, 75, 90, 170, 802
百位: 0   0   0   0   0   0   1    8
```

百位為 0 的那群（2~90）沿用第 2 趟已排好的順序 → 完成 `[2, 24, 45, 66, 75, 90, 170, 802]`。

**關鍵**：若任一趟的排序不穩定，前一趟辛苦排好的「低位順序」就會被打亂——這就是為什麼底層必須用 Counting Sort（穩定）而非 Quick Sort。

## 關鍵洞察

正確性依賴**每一位的排序必須是穩定的**。若某一位的排序不穩定，前幾位的排序結果會被破壞。此處的 `countingSortByDigit` 用「前綴和 + 反向遍歷」保證穩定。

## 前提條件

- 元素為**非負整數**（此處未做 `min` 偏移）。
- ⚠️ 違反時：負數的位數計算錯誤，會**靜默產生含 null 的錯誤結果**（不會報錯）。需排序負數請改用 [Counting Sort](../counting-sort/README.md)（已內建 min 偏移）或先自行偏移。

## 複雜度

| Best        | Average     | Worst       | 空間     | 穩定 | In-place |
| ----------- | ----------- | ----------- | -------- | ---- | -------- |
| O(d(n + k)) | O(d(n + k)) | O(d(n + k)) | O(n + k) | ✅   | ❌       |

d = 最大位數，k = 基數（此處 k = 10）；d 小時趨近 O(n)。

## 常見陷阱

- **底層用不穩定排序**（如對每一位用 Quick Sort）→ 破壞前幾位的排序結果。必須用穩定排序。
