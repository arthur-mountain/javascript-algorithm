# Binary Search

## 🎯 設計理念

### 存在意義

**這個演算法策略被發明出來，是為了解決什麼類型的問題？**

- **暴力法**：線性掃描整個搜尋空間，逐一檢查每個元素是否為目標 → O(n)。瓶頸在於每次比較只排除一個元素，完全沒有利用搜尋空間的結構。
- **可利用的結構特性**：搜尋空間具有**單調性**（monotonicity）——對某個條件 `condition(x)` 而言，存在一個分界點，使得分界點一側全部滿足、另一側全部不滿足。因此，只要在任意一點評估條件，就能確定答案在哪一半。
- **核心加速原理**：每次比較排除一半搜尋空間，將 O(n) 降至 O(log n)。

### 心智模型

**類比：猜數字遊戲**

對方心裡想一個 1 到 100 的數字，你每猜一次，對方會告訴你「太大」或「太小」。最佳策略是永遠猜中間值——第一次猜 50，如果「太大」就知道答案在 1\~49，接著猜 25，依此類推。每一輪排除一半的可能性，最多 7 次（⌈log₂ 100⌉）就能猜到。

**局限性**：猜數字的場景假設了完美的二分切割和 O(1) 的比較成本。在「答案空間二分」（Binary Search on Answer）的變體中，每次「比較」實際上是一次 O(n) 或更高成本的驗證函式呼叫，因此總複雜度不是 O(log n) 而是 O(log R × C)，其中 R 是答案值域，C 是單次驗證成本。

## 🏗️ 運作機制

### 核心組件

| 組件         | 功能                  | 為什麼需要                                     |
| ------------ | --------------------- | ---------------------------------------------- |
| 左邊界 `lo`  | 搜尋範圍的下界        | 定義搜尋空間的左側，每次迭代可能右移以縮小範圍 |
| 右邊界 `hi`  | 搜尋範圍的上界        | 定義搜尋空間的右側，每次迭代可能左移以縮小範圍 |
| 中間點 `mid` | 當前迭代的決策點      | 在搜尋空間正中間取樣，根據評估結果決定搜尋方向 |
| 判斷條件     | 決定 mid 與目標的關係 | 核心邏輯——決定搜尋空間應該往左半還是右半縮小   |

### 狀態表示

- **需要維護的變數**：`lo`、`hi`（搜尋範圍的左右邊界）
- **初始狀態**：`lo` 和 `hi` 都是可能出現答案的位置。具體初始值由題目決定——答案只可能在索引 `0` 到 `n-1` 時，`hi = n - 1`；位置 `n` 也是合法答案時（如插入位置），`hi = n`
- **不變量（Invariant）**：答案（如果存在）始終位於 `[lo, hi]` 範圍內
- **終止條件**：`lo === hi`，搜尋空間收斂到唯一位置

### 適用前提條件

**使用這個策略之前，必須確認以下前提條件成立：**

| 前提條件                 | 為什麼需要                                                                                          | 違反時的後果                                                                            | 替代方案                                                       |
| ------------------------ | --------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------- | -------------------------------------------------------------- |
| 搜尋空間具有**單調性**   | 需要根據 mid 的評估結果，確定性地排除一半空間。單調性保證了「一側全滿足、另一側全不滿足」的分界結構 | 無法判斷答案在左半還是右半 → 可能排除包含答案的區域 → 得到**錯誤結果**                  | 線性掃描 O(n)、HashMap O(n)、排序後再二分 O(n log n)           |
| 搜尋空間有**明確邊界**   | `lo` 和 `hi` 必須有具體的初始值才能啟動演算法                                                       | 無法初始化搜尋空間 → 演算法無法開始                                                     | 使用指數擴展（Exponential Search）先找到上界，再對該範圍做二分 |
| 搜尋空間可被**隨機存取** | 需要在 O(1) 內存取 mid 位置的值                                                                     | 若 mid 存取需 O(n)（如 LinkedList），則每次迭代 O(n)，總共 O(n log n)，不如直接線性掃描 | 轉換為 Array 後再二分，或直接使用線性掃描                      |

### 與資料結構的關聯

底層所依賴的資料結構來儲存資料與維護狀態：

| 資料結構                        | 角色     | 在本演算法中的用途                                          | 典型題目/場景                                                 |
| ------------------------------- | -------- | ----------------------------------------------------------- | ------------------------------------------------------------- |
| Sorted Array                    | 輸入載體 | 儲存已排序的元素，提供 O(1) 隨機存取                        | LC 704 Binary Search、LC 34 Find First and Last Position      |
| 答案值域（隱式整數/浮點數範圍） | 輸入載體 | 不操作具體陣列，而是在一段數值範圍上搜尋                    | LC 875 Koko Eating Bananas、LC 1011 Capacity To Ship Packages |
| 驗證函式 `feasible(mid)`        | 狀態維護 | 在答案空間二分中，取代直接比較，用一個函式判斷 mid 是否可行 | LC 410 Split Array Largest Sum                                |

**關鍵洞察**：Binary Search 定義「如何根據 mid 的評估結果縮小搜尋空間」的邏輯，底層資料結構（Array 或隱式數值範圍）負責提供可被二分的搜尋空間。選擇哪種輸入載體，取決於搜尋空間是顯式的（陣列索引）還是隱式的（答案值域）。

### 流程步驟——從不變量推導一切

Binary Search 的所有實作細節都可以從**一句不變量**推導出來，不需要獨立記憶任何規則。以下是完整的推導鏈。

---

**Step 0：定義不變量（Invariant Definition）**

> **不變量：「第一個滿足 condition 的位置」一定在 `[lo, hi]` 範圍內。**

這是整個 Binary Search 唯一需要記住的一句話。後面的所有決定——while 條件、邊界更新、mid 計算、返回值——全部是這句話的邏輯必然結果。

---

**Step 1：從不變量推導 while 條件**

**目的**：判斷搜尋是否結束。

**推導方法**：問自己——「`lo === hi` 時，我還需要做什麼嗎？」

答案取決於搜尋的目標是什麼：

**邊界搜尋（findFirst / findLast）**：目標是找到某個「位置」。`lo === hi` 時，位置已被鎖定到唯一候選，不需要再進迴圈——迴圈外直接 `return lo` 就是答案。

```javascript
while (lo < hi)  // lo === hi 時位置已鎖定，搜尋結束
```

**精確搜尋（exactSearch）**：目標是找到一個「值等於 target 的位置」。`lo === hi` 時，位置鎖定了，但**那個位置上的值是否等於 target 還沒驗證過**。所以有兩種處理方式：

```javascript
// 方式一：讓 lo === hi 也進入迴圈，在迴圈內驗證
while (lo <= hi)   // lo === hi 時還有一個值需要檢查

// 方式二：lo === hi 不進迴圈，在迴圈外做後處理
while (lo < hi)    // 迴圈後：return nums[lo] === target ? lo : -1
```

兩種都正確，本文的精確搜尋採用方式一。

**核心原則不變**：while 條件永遠是在問「根據不變量，此時還有未完成的工作嗎？」邊界搜尋的「工作」是縮小位置範圍，`lo === hi` 時工作完成；精確搜尋的「工作」包含驗證值，`lo === hi` 時還有一次驗證要做。

---

**Step 2：從不變量推導邊界更新（Constraint Satisfaction）**

**目的**：評估 `condition(mid)` 後，縮小搜尋空間，但**不能破壞不變量**。

**推導**：

**情況 A：`condition(mid)` 為 true**

mid 可能是答案（第一個 T 可能就是 mid，也可能在更左邊）。不變量要求答案在 `[lo, hi]` 內，所以新區間**必須包含 mid**。

```javascript
hi = mid; // 新區間 [lo, mid] — 保留 mid，不違反不變量 ✓
```

如果寫 `hi = mid - 1`，就把 mid 排除了——但 mid 可能是答案，**違反不變量** ✗。

**情況 B：`condition(mid)` 為 false**

mid 確定不是答案。不變量要求答案在 `[lo, hi]` 內，而 mid 不是答案，所以新區間**不需要包含 mid**。

```javascript
lo = mid + 1; // 新區間 [mid + 1, hi] — 排除 mid，不違反不變量 ✓
```

如果寫 `lo = mid`（保留一個已知不是答案的位置），雖然不違反不變量，但搜尋空間可能不縮小 → 會造成無限迴圈（見 Step 3）。

**核心原則**：每次更新都是在問「這個操作會不會把答案排除在 `[lo, hi]` 之外？」如果會，就不能做。

---

**Step 3：從不變量推導 mid 計算**

**目的**：在 `[lo, hi]` 中選一個中間點。

**推導**：

```javascript
const mid = lo + Math.floor((hi - lo) / 2);
```

為什麼用 `Math.floor`（偏左）？因為 Step 2 決定了更新方式是 `hi = mid` 和 `lo = mid + 1`。驗證最危險的邊界情況 `lo + 1 === hi`：

- `mid = lo + Math.floor(1 / 2) = lo`（偏左 → mid === lo）
- 走 `hi = mid` → `hi` 從 `lo + 1` 變成 `lo` → 搜尋空間縮小 ✓
- 走 `lo = mid + 1` → `lo` 從 `lo` 變成 `lo + 1` → 搜尋空間縮小 ✓

兩條路都能縮小搜尋空間 → 不會無限迴圈。

如果反過來用 `Math.ceil`（偏右），`mid === hi`，走 `hi = mid` 時 `hi` 不會改變 → 無限迴圈。所以 `Math.floor` 不是「規定」，是從 Step 2 的更新方式**推導出的唯一安全選擇**。

（`lo + Math.floor((hi - lo) / 2)` 而非 `Math.floor((lo + hi) / 2)` 是為了避免 `lo + hi` 超過 Number.MAX_SAFE_INTEGER。）

---

**Step 4：結算（Post-Condition Update）**

**目的**：迴圈結束時提取答案。

**推導**：迴圈結束時 `lo === hi`。不變量保證「答案在 `[lo, hi]` 內」，而 `lo === hi` 意味著答案被鎖定在這唯一位置。

```javascript
return lo; // 第一個滿足 condition 的位置
```

---

**推導鏈總結**

```plaintext
不變量：答案在 [lo, hi] 內
    ↓
while 條件：
  邊界搜尋 → lo < hi（lo === hi 時位置已鎖定，不需再處理）
  精確搜尋 → lo <= hi（lo === hi 時還需驗證值是否等於 target）
    ↓
邊界更新：condition(mid) 為 true → hi = mid（保留 mid，否則違反不變量）
          condition(mid) 為 false → lo = mid + 1（排除 mid，安全不違反）
    ↓
mid 計算：Math.floor（偏左，搭配 hi = mid / lo = mid + 1 不會死迴圈）
    ↓
結算：
  找第一個 T → return lo
  找最後一個 T → return lo - 1（lo 指向第一個 F，退一格）
  精確搜尋 → 迴圈內直接 return mid，或迴圈結束 return -1
```

**每一行都是上一行的邏輯必然結果，沒有任何規則需要獨立記憶。**

### 模式分類

**統一原則**：所有 Binary Search 都只做一件事——「找第一個滿足 condition 為 true 的位置」。上面的推導鏈適用於所有變體，差異只在 condition 的定義和結算方式。

**變體總覽**：

```plaintext
Binary Search（統一為「找第一個 T」）
├── 精確搜尋（唯一例外：用三路比較 + 提前 return）
├── 找第一個 T（推導鏈的原生形式，直接套用）
├── 找最後一個 T（語義轉換：condition 取反 → 找第一個 F → 減 1）
└── 答案空間二分（搜尋空間從陣列索引換成數值範圍，其餘不變）
```

---

**變體 A：精確搜尋（Exact Search）**

**觸發條件**：題目明確要求「找到 target 並返回索引」，找不到返回 -1。

**為什麼是例外（從不變量推導）**：

精確搜尋的不變量同樣是「答案在 `[lo, hi]` 內」，但它的「工作」多了一步——不只要鎖定位置，還要驗證 `nums[mid] === target`。這導致兩個推導結果不同：

1. **while 條件用 `<=`**：`lo === hi` 時位置鎖定了，但那個值還沒被驗證過。根據 Step 1 的推導——「還有未完成的工作嗎？」→ 有（驗證）→ 需要進入迴圈。
2. **兩側都用 `mid ± 1`**：三路比較中，`nums[mid] === target` 會直接 return；走到 `nums[mid] < target` 或 `nums[mid] > target` 時，mid **確定不是答案**。既然確定不是答案，兩側都可以安全排除 mid（`lo = mid + 1` 和 `hi = mid - 1`），不違反不變量。

這兩個差異都不是「例外規則」，而是同一個不變量在不同目標下的推導結果。

**不變量**：答案（如果存在）在 `[lo, hi]` 內。

```javascript
function exactSearch(nums, target) {
  let lo = 0;
  let hi = nums.length - 1;

  // lo === hi 時那個元素可能就是 target，還需要檢查 → 用 <=
  while (lo <= hi) {
    const mid = lo + Math.floor((hi - lo) / 2);

    if (nums[mid] === target) {
      return mid;
    } else if (nums[mid] < target) {
      // mid 確定不是答案 → 排除 mid（不違反不變量）
      lo = mid + 1;
    } else {
      // mid 確定不是答案 → 排除 mid（不違反不變量）
      hi = mid - 1;
    }
    // 兩種更新都在排除 mid，所以用 +1 / -1
  }

  return -1;
}
```

---

**變體 B：找第一個 T（Find First / Lower Bound）——推導鏈的原生形式**

**觸發條件**：搜尋空間呈現 `[F F F F T T T T]` 結構，要找第一個 T。

**這就是推導鏈本身**，沒有任何額外規則。

**不變量**：第一個 T 在 `[lo, hi]` 內。

```javascript
function findFirst(lo, hi, condition) {
  // 不變量：第一個 T 在 [lo, hi] 內
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);

    if (condition(mid)) {
      // mid 是 T → 保留 mid（排除它違反不變量）
      hi = mid;
    } else {
      // mid 是 F → 排除 mid（它確定不是答案）
      lo = mid + 1;
    }
  }

  // lo === hi，答案鎖定
  return lo;
}
```

---

**變體 C：找最後一個 T（Find Last）——語義轉換**

**觸發條件**：搜尋空間呈現 `[T T T T F F F F]` 結構，要找最後一個 T。

**語義轉換**：把「找最後一個 T」轉化為「找第一個 F，然後減 1」。轉換後搜尋空間變成 `[F F F F T T T T]`（condition 取反），就能**完全複用推導鏈**——骨架完全不變，只改兩個地方：condition 取反、return lo - 1。

**不變量**：第一個 F 在 `[lo, hi]` 內。（hi 通常需設為 n 而非 n-1，作為哨兵。）

```javascript
function findLast(lo, hi, condition) {
  // 不變量：第一個 F 在 [lo, hi] 內
  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);

    // 與 findFirst 的唯一差異：condition 取反
    if (!condition(mid)) {
      hi = mid;
    } else {
      lo = mid + 1;
    }
  }

  // lo 指向第一個 F；lo - 1 即為最後一個 T
  return lo - 1;
}
```

---

**變體 D：答案空間二分（Binary Search on Answer）**

**觸發條件**：題目問「某個值的最小/最大可能是多少」，答案值域有單調性。

**與陣列二分的差異**：只有一個——搜尋空間從陣列索引換成數值範圍。推導鏈完全不變。

**不變量**：答案在 `[lo, hi]` 內。

```javascript
// 找最小可行值（FFFTTT → 直接用 findFirst）
function binarySearchOnAnswerMin(minVal, maxVal, feasible) {
  let lo = minVal;
  let hi = maxVal;

  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (feasible(mid)) {
      hi = mid;
    } else {
      lo = mid + 1;
    }
  }

  return lo;
}

// 找最大可行值（TTTFFF → 用 findLast：取反 + 減 1）
function binarySearchOnAnswerMax(minVal, maxVal, feasible) {
  let lo = minVal;
  let hi = maxVal;

  while (lo < hi) {
    const mid = lo + Math.floor((hi - lo) / 2);
    if (!feasible(mid)) {
      hi = mid;
    } else {
      lo = mid + 1;
    }
  }

  return lo - 1;
}
```

### 複雜度總結

| 變體         | 時間複雜度   | 空間複雜度 | 備註                               |
| ------------ | ------------ | ---------- | ---------------------------------- |
| 精確搜尋     | O(log n)     | O(1)       | 可能提前 return                    |
| 找第一個 T   | O(log n)     | O(1)       | 固定 ⌈log₂ n⌉ 次迭代               |
| 找最後一個 T | O(log n)     | O(1)       | condition 取反 + return lo - 1     |
| 答案空間二分 | O(log R × C) | O(1)       | R = 答案值域大小，C = 驗證函式成本 |

**符號定義**：

- `n`：搜尋空間的元素數量
- `R`：答案值域大小（`maxVal - minVal`）
- `C`：驗證函式 `feasible(mid)` 的時間成本

## ⭐ 抽象化翻譯器

**這是最關鍵的部分——建立「問題情境 → 抽象工具」的映射能力**

### 識別核心抽象

> 這個工具的核心對象是：**搜尋空間**（一段有序的、可被二分的範圍）
>
> 它管理的是這些對象之間的什麼關係？**透過單調性判斷，反覆將搜尋空間砍半，直到收斂到目標位置或目標值**

### 建立映射維度

| 維度     | 要回答的問題                                                              | 這個答案決定了什麼                                             |
| -------- | ------------------------------------------------------------------------- | -------------------------------------------------------------- |
| 搜尋空間 | 二分的範圍是什麼？是陣列索引 `[0, n-1]` 還是答案值域 `[minVal, maxVal]`？ | 決定 lo, hi 的初始值                                           |
| 單調性   | 搜尋空間上的什麼性質是單調的？分界結構是 `FFFTTT` 還是 `TTTFFF`？         | FFFTTT → 直接套推導鏈；TTTFFF → condition 取反 + return lo - 1 |
| 判斷條件 | `condition(mid)` 具體是什麼？                                             | 推導鏈中唯一需要根據題目填入的部分                             |
| 答案提取 | 迴圈結束後 return lo 還是 lo - 1？是否需要邊界檢查？                      | 找第一個 T → return lo；找最後一個 T → return lo - 1           |

### 實戰檢查表

```plaintext
題目:_______________

1. 搜尋空間 [lo, hi]：_______________
2. 單調性結構：FFFTTT / TTTFFF？
3. condition(mid) = _______________
4. 答案提取：return lo / return lo - 1

→ FFFTTT：直接套推導鏈
→ TTTFFF：condition 取反 + return lo - 1
```

### 映射範例

**題目 A**：LC 875 Koko Eating Bananas

> Koko 有 n 堆香蕉，第 i 堆有 piles[i] 根。警衛 h 小時後回來。求最小的吃速 k 使得 Koko 能在 h 小時內吃完。

| 維度     | 具體問題                              | 抽象映射                   |
| -------- | ------------------------------------- | -------------------------- |
| 搜尋空間 | k 的範圍 `[1, max(piles)]`            | lo = 1, hi = max(piles)    |
| 單調性   | k 越大越容易吃完 → `FFFTTT`           | 直接套推導鏈（找第一個 T） |
| 判斷條件 | `feasible(k)` = Σ⌈piles[i] / k⌉ ≤ h？ | condition = feasible       |
| 答案提取 | return lo                             | 不需後處理                 |

**題目 B**：LC 34 Find First and Last Position of Element in Sorted Array

> 給定升序陣列 nums 和 target，找 target 第一次和最後一次出現的索引。

| 維度       | 具體問題                                         | 抽象映射                                     |
| ---------- | ------------------------------------------------ | -------------------------------------------- |
| 搜尋空間   | 陣列索引 `[0, n]`（n 是哨兵）                    | lo = 0, hi = n                               |
| 找第一次   | `nums[mid] >= target` → `FFFTTT`                 | findFirst：condition = `nums[mid] >= target` |
| 找最後一次 | `nums[mid] <= target` → `TTTFFF`                 | findLast：condition 取反 + return lo - 1     |
| 答案提取   | 需檢查 `nums[first] === target`，否則 `[-1, -1]` | 需後處理                                     |

## 🔍 觸發器（模式識別）

### 正向觸發器

| 層級     | 特徵                                                              | 為什麼這個特徵指向 Binary Search                    |
| -------- | ----------------------------------------------------------------- | --------------------------------------------------- |
| 關鍵字   | "sorted array"、"search"、"find minimum/maximum speed/capacity"   | 排序 = 天然單調性；最佳化問題常可轉化為答案空間二分 |
| 結構     | 輸入已排序，或問題空間具有 `FFFTTT` / `TTTFFF` 的分界結構         | Binary Search 的本質就是找這個分界點                |
| 目標     | 「找特定值」「找第一個/最後一個」「最小化最大值」「最大化最小值」 | 這四類目標直接對應四個變體                          |
| 操作模式 | 每次可以根據某個判斷排除一半的搜尋空間                            | 這正是 Binary Search 的核心操作                     |
| 約束     | n ≤ 10⁸ 或更大，且暴力法 O(n) 可能超時 → 需要 O(log n)            | log₂(10⁸) ≈ 27 次迭代，極其高效                     |

**一句話觸發規則**：當搜尋空間有序（或具有單調性），且每次判斷能確定性地排除一半時，使用 Binary Search。

### 反向觸發器

| 陷阱情境             | 為什麼不適用                                                    | 更好的選擇                                       |
| -------------------- | --------------------------------------------------------------- | ------------------------------------------------ |
| 未排序的陣列找特定值 | 沒有單調性，無法判斷往哪邊搜尋                                  | HashMap O(n) 查找；或先排序再二分                |
| Rotated Sorted Array | 乍看不適用，但實際上**仍可用**——每次 mid 都能判斷哪一半是有序的 | 仍然用 Binary Search，但判斷邏輯需額外處理旋轉點 |
| 找所有滿足條件的元素 | Binary Search 只能定位分界點，不能列舉所有結果                  | 找到邊界後用線性掃描                             |
| 可行性不具單調性     | 例如 `feasible(5)=T, feasible(6)=F, feasible(7)=T`              | DP、Backtracking、線性掃描                       |

### 與類似工具的決策點

```plaintext
在有序空間中搜尋
├── 找特定值是否存在 → Binary Search 精確搜尋（O(log n)）
├── 找特定值但資料動態變化 → Balanced BST / HashSet
├── 找第一個/最後一個滿足條件的位置 → Binary Search 邊界搜尋（O(log n)）
├── 找第 k 大/第 k 小 → Quick Select O(n) / Heap O(n log k)
├── 最佳化問題且具單調性 → Binary Search on Answer（O(log R × C)）
└── 最佳化問題但不具單調性 → DP / Greedy
```

## ⚠️ 注意事項

### 常見陷阱

| 陷阱                          | 錯誤寫法                          | 正確寫法                         | 為什麼（從不變量推導）                                                            |
| ----------------------------- | --------------------------------- | -------------------------------- | --------------------------------------------------------------------------------- |
| 整數溢位                      | `Math.floor((lo + hi) / 2)`       | `lo + Math.floor((hi - lo) / 2)` | `lo + hi` 可能超過 Number.MAX_SAFE_INTEGER                                        |
| condition 為 true 時排除 mid  | `hi = mid - 1`                    | `hi = mid`                       | mid 可能是答案（第一個 T），排除它**違反不變量**                                  |
| condition 為 false 時保留 mid | `lo = mid`                        | `lo = mid + 1`                   | mid 確定不是答案（它是 F），保留它雖不違反不變量，但搜尋空間可能不縮小 → 無限迴圈 |
| findLast 忘記減 1             | `return lo`                       | `return lo - 1`                  | lo 指向第一個 F，最後一個 T 在它左邊一格                                          |
| findLast 的 hi 初始值設太小   | `hi = n - 1`                      | `hi = n`（哨兵）                 | 全部都是 T 時，「第一個 F」在位置 n；如果 hi = n-1 會遺漏                         |
| 邊界搜尋忘記後處理            | 直接 `return lo` 假設答案一定存在 | `return condition(lo) ? lo : -1` | 所有元素都是 F 時，lo 會停在哨兵位置                                              |

### 常見變體

| 變體             | 修改內容                                         | 適用場景                 |
| ---------------- | ------------------------------------------------ | ------------------------ |
| 旋轉陣列上的二分 | Evaluate 步驟增加判斷「哪一半有序」的邏輯        | LC 33, 81, 153           |
| 二維矩陣二分     | 將 2D `(row, col)` 映射為 1D `mid`               | LC 74 Search a 2D Matrix |
| 浮點數二分       | 迴圈條件改為 `hi - lo > epsilon` 或固定迭代次數  | 求平方根、幾何問題       |
| Peak Finding     | 比較 `nums[mid]` 與 `nums[mid + 1]` 判斷搜尋方向 | LC 162, LC 852           |
