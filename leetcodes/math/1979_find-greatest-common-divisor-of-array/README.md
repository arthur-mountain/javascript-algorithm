## 問題描述

從輸入陣列中，找到最大、最小值中間的最大的公因數

## Constraints

- 2 <= nums.length <= 1000
- 1 <= nums[i] <= 1000

## 解題思路

### 初步分析(觀察與發想)

- 需要先知道範圍，需要一次遍歷找到陣列中的最大、最小值

- 有了最大、最小值後，從最小值遞減遍歷（因為 GCD 不可能大於兩數中較小的那一個），找到最大公因數可以整除最大、最小值

  - 候選公因數不可能大於兩數中較小的那一個（也就是 min），所以搜尋範圍可以直接鎖定在 [1, min] 之間

## 解法總覽

### Solution1：

- **核心策略**：

  先找到最大、最小值，從最小值遞減遍歷找到最大公因數

- **思路說明**：

  1. 先遍歷輸入 nums，找到最大、最小值

  2. 因為候選公因數不可能大於兩數中較小的那一個（min），所以只需要在 [1, min] 範圍內搜尋；為了找到「最大」公因數，必須從 min 開始往下遞減檢查，第一個能同時整除 min、max 的數字，就是答案

  補充：fallback 的最大公因數是 1（兩數互質時的保底解）

- **複雜度分析**：

  - 時間複雜度：O(n + min(nums))（找最大最小值 O(n) + 從 min 遞減搜尋，最壞情況（min 與 max 互質時）要掃到 i = 1，耗費 O(min(nums))）

  - 空間複雜度：O(1)

  - 通過狀態：✅ AC

- **其他備註\(優化方向、特殊限制、問題延伸討論\)**：

  - 找到最大公因數那段，應該有數學公式，而非透過遍歷遞減找公因數

- **測試案例**：

  - 案例 A：最小長度陣列 — 極小輸入下界

    Input: nums = [2, 4]

    Ouput: 2

    Expected: 2

    Explain: nums.length 下界為 2，驗證兩元素時 min/max 初始化（min = max = nums[0]）後，只需再比較一次 nums[1] 即可正確運作

  ***

  - 案例 B：min = 1 — 值域下界觸發提前結束

    Input: nums = [1, 1000]

    Ouput: 1

    Expected: 1

    Explain: i 從 min = 1 開始遞減，第一次迭代 `min % 1 === 0 && max % 1 === 0` 即成立，立即 return，此案例為 O(1) 的最好情況（提前終止），並非掃到底的最壞情況

  ***

  - 案例 C：互質無字面 1 — fallback 真實觸發路徑

    Input: nums = [8, 3, 5]

    Ouput: 1

    Expected: 1

    Explain: min = 3, max = 8

### Solution2：

- **核心策略**：

  改用歐幾里得演算法（輾轉相除法），取代 Solution1 逐一遞減找因數的方式

  一句話解釋：因為任何能同時整除 a、b 的數，也一定能整除 a % b，所以問題可以換成更小的一組數，答案不變。

- **思路說明**：

  1. 一樣先遍歷 nums，找出最大值、最小值

  2. 用輾轉相除法：設 a = max、b = min，重複執行「a, b = b, a % b」，直到 b === 0，此時 a 即為兩數的 GCD

     （原理：gcd(a, b) = gcd(b, a % b)，因為任何能同時整除 a、b 的數，也一定能整除 a % b，所以問題可以不斷縮小成更小的一組數對，直到其中一個變成 0）

  3. 相較 Solution1 用遞減方式一個一個嘗試因數，這裡用取餘數的方式跳躍式縮小數值範圍，收斂速度快很多

- **複雜度分析**：

  - 時間複雜度：O(n + log(min(nums)))（找最大最小值 O(n) + 輾轉相除法 O(log(min))）

  - 空間複雜度：O(1)

  - 通過狀態：✅ AC

- **其他備註\(優化方向、特殊限制、問題延伸討論\)**：

  - 這就是 Solution1 備註中提到「應該有數學公式」的那個解法，用輾轉相除法取代暴力遞減

  - 輾轉相除法的收斂速度是 O(log(min(a, b)))，遠快於 Solution1 最壞情況（min = 1 時）退化成的 O(max)

- **測試案例**：同 Solution1

### Solution3：

- **核心策略**：

  歐幾里得演算法的遞迴寫法，邏輯與 Solution2 完全相同，差別只在用遞迴取代 while 迴圈

- **思路說明**：

  1. 一樣先找出最大值、最小值

  2. 定義遞迴函式 gcd(a, b)：若 b === 0，回傳 a（base case）；否則回傳 gcd(b, a % b)

  3. 與 Solution2 是同一套輾轉相除法邏輯，差別只在於用遞迴呼叫自己來逼近 base case，而不是用 while 迴圈手動迭代

- **複雜度分析**：

  - 時間複雜度：O(n + log(min(nums)))（與 Solution2 相同）

  - 空間複雜度：O(log(min(nums)))（遞迴呼叫堆疊深度等於輾轉相除法的迭代次數，這是與 Solution2 唯一的複雜度差異）

  - 通過狀態：✅ AC

- **其他備註\(優化方向、特殊限制、問題延伸討論\)**：

  - 本題數值上限僅 1000，遞迴深度非常淺，不會有 stack overflow 風險，但仍是與 Solution2 在空間複雜度上的明確差異點，面試時若被追問通常要能講出這個 trade-off

  - Follow-up 方向：若輸入數值上限拉到極大（如 10^9 以上），迭代版本（Solution2）在空間上會更穩健

- **測試案例**：同 Solution1
