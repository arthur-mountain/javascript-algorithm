# Monotonic Queue（單調佇列）

## 🎯 設計理念

### 存在意義

**這個工具被發明出來，是為了解決什麼類型的問題？**

- **在它出現之前的痛點**：在「滑動窗口」場景中，每移動一步就需要找窗口內的最大/最小值。

  - 暴力法每次掃描整個窗口，對長度為 `k` 的窗口需要 O(k)，整體 O(n × k)。
  - 使用 Heap（Priority Queue）可以做到 O(n log k)，但無法高效移除過期元素（Heap 的 delete 需要額外定位）。

- **核心價值主張**：以 **O(1) 攤銷**（攤銷 amortized：單次最壞可能 O(n)，但整段操作的總成本有上界，平均下來每次仍是 O(1)，完整論證見「複雜度總結」）完成「加入新元素」與「查詢當前窗口極值」，使整體滑動窗口問題達到 **O(n)**。

- **一句話本質**：一個自動淘汰「不可能成為答案的候選」的有序雙端佇列。

```plaintext
痛點：滑動窗口求極值，暴力 O(n×k)，Heap O(n log k) 且移除過期元素困難
價值主張：每個元素最多進出佇列一次，整體 O(n)
一句話本質：一個自動淘汰「不可能成為答案的候選」的有序雙端佇列（Deque）
```

### 心智模型

**用生活化的類比建立直覺**

**類比：排隊等候的身高淘汰賽**

想像一排人站在觀景台前等著看風景，規則是「只有前方沒有比你高的人擋住時，你才有機會看到風景」：

- **Push（新人加入）**：一個新人站到隊伍最後面。他先看排在他前面（也就是比他早到、站得比他前面）的人——如果緊鄰他前面有人比他矮，那些人「永遠不可能比他先看到風景」（因為他比較高、又比較晚離開），所以那些矮的人直接離隊。一直淘汰到遇到比他高的人為止。
- **Front（查詢最高的人）**：隊伍最前面的人永遠是最高的（因為所有比他矮又比他早來的人，都被後來更高的人淘汰了）。
- **PopExpired（前面的人離開窗口）**：最前面的人時間到了要離場，直接從前面移除。

**局限性**：

- 真實排隊的人不會因為後面來了更高的人就離開；但 Monotonic Queue 會主動淘汰「被支配的候選」。
- 類比中「高矮」是固定的；但在某些應用中，「支配關係」可以由自定義比較函式決定（不限於大小比較）。

## 🏗️ 抽象結構

### 核心組件

| 組件          | 功能                                                                          | 為什麼需要                                                                             |
| ------------- | ----------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| `deque`       | 底層儲存結構（陣列），存放**索引**                                            | 索引同時承載兩個資訊：透過索引查原陣列得到「值」用於比較，索引本身用於判斷「是否過期」 |
| `nums`        | 原始輸入陣列的參考                                                            | 比較時需要查詢 `nums[index]` 取得實際值                                                |
| `shouldEvict` | 淘汰判定函式 `(backVal, newVal) => boolean`，回傳 `true` 表示尾端元素應被淘汰 | 決定單調方向：遞減佇列（求最大）或遞增佇列（求最小）                                   |
| `head`        | 前端指標（整數）                                                              | 以移動指標取代 `Array.shift()`，避免 O(n) 搬移；搭配定期壓縮控制物理空間               |

### 狀態表示

- **使用什麼底層資料結構**：JavaScript 原生陣列，以 `push()` / `pop()` 操作尾端，以索引指標 `head` 操作前端（避免 `shift()` 的 O(n) 搬移成本），並在前端垃圾累積過多時一次性壓縮回收。

- **初始狀態**：

  ```javascript
  deque = []; // 空佇列，無候選者
  head = 0; // 前端指標指向起始位置
  ```

- **關鍵狀態的定義規則**：
  - **空佇列**：`head >= deque.length`
  - **佇列有效範圍**：`deque[head]` 到 `deque[deque.length - 1]`
  - **單調不變量（Monotonic Decreasing Queue，求最大）**：對所有有效索引 `i < j`，`nums[deque[i]] >= nums[deque[j]]`
  - **單調不變量（Monotonic Increasing Queue，求最小）**：對所有有效索引 `i < j`，`nums[deque[i]] <= nums[deque[j]]`

### 核心操作

#### 1. push(index)

**目的**：將新索引加入佇列尾端，同時維持單調性——淘汰所有「被支配的候選」。

**輸入/輸出**：

- 輸入：新元素的索引 `index`
- 輸出：無（修改內部狀態）

**不變量**：執行後，`deque` 中的有效元素仍滿足單調性；新加入的索引永遠在最尾端。

**基礎實作**：

```javascript
// 最直覺的寫法：直接加到尾端，不做任何淘汰
push(index) {
  this.deque.push(index);
}
```

**基礎版本的問題**：

- 佇列中混雜了「不可能成為答案的元素」，查詢極值需要掃描整個佇列 → O(k)。
- 佇列大小持續增長，沒有自動清理機制。

**優化策略**：尾端淘汰（Back Elimination）

- 核心思想（一句話）：新元素加入前，從尾端移除所有「被新元素支配」的舊元素。
- 為什麼這樣能解決問題？以 max queue 為例，如果一個舊元素的值不如新元素（值較小），且舊元素比新元素更早過期（索引更小），那麼這個舊元素在任何未來的窗口中都不可能是答案——只要舊元素還在窗口內，更晚過期、值又更大的新元素必然也在窗口內，舊元素永遠被壓著。所以可以安心丟棄。

```javascript
push(index) {
  // 從尾端淘汰所有被新元素支配的候選
  // 以 Monotonic Decreasing Queue 為例：移除所有值 <= nums[index] 的尾端元素
  while (
    this.head < this.deque.length &&
    this.nums[this.deque[this.deque.length - 1]] <= this.nums[index]
  ) {
    this.deque.pop();
  }
  this.deque.push(index);
}
```

**視覺化說明**：

```plaintext
Monotonic Decreasing Queue（尾端淘汰過程）
nums = [3, 1, 4, 1, 5]

push(0)  → deque: [0]         值: [3]
push(1)  → deque: [0, 1]      值: [3, 1]     ← 1 < 3，不淘汰，直接加入
push(2)  → deque: [2]         值: [4]         ← 4 > 1，淘汰 index 1；4 > 3，淘汰 index 0；加入
push(3)  → deque: [2, 3]      值: [4, 1]     ← 1 < 4，不淘汰，直接加入
push(4)  → deque: [4]         值: [5]         ← 5 > 1，淘汰 index 3；5 > 4，淘汰 index 2；加入

觀察：
- 每個元素最多被 push 一次、被 pop 一次 → 5 個元素總共最多 10 次操作
- 淘汰邏輯只看「值」：舊元素值 ≤ 新元素值 → 舊元素永遠不可能是答案，淘汰
- 窗口邊界的過期移除不在這裡處理，那是 popExpired 的職責
```

**優化後的複雜度**：單次 O(n) 最壞（全部淘汰），但 **O(1) 攤銷**（每個元素最多被 push 一次、pop 一次，n 個元素總共最多 2n 次操作）。

#### 2. popExpired(leftBound)

**目的**：移除佇列前端所有已「過期」（索引超出當前窗口左邊界）的元素。

**輸入/輸出**：

- 輸入：當前窗口的左邊界 `leftBound`（inclusive；索引 `< leftBound` 的元素視為過期）
- 輸出：無（修改內部狀態）

**不變量**：執行後，佇列前端有效元素的索引 `>= leftBound`。

**基礎實作**：

```javascript
// 使用 Array.shift() 移除前端
popExpired(leftBound) {
  while (this.deque.length > 0 && this.deque[0] < leftBound) {
    this.deque.shift(); // O(n)：所有元素左移一位
  }
}
```

**基礎版本的問題**：

- `Array.shift()` 需要將後續所有元素向前搬移一位 → 單次 O(n)。
- 在滑動窗口中頻繁呼叫，累積成本可能顯著。

**優化策略**：前端指標（Head Pointer）＋ 定期壓縮（Compaction）

- 核心思想（一句話）：不真正移除前端元素，而是移動 `head` 指標跳過過期元素（解決**時間**）；當「被跳過的垃圾」累積到不少於「有效元素」時，再一次性把前段切掉以回收記憶體（解決**空間**）。
- 為什麼這樣能解決問題？
  - **時間**：邏輯上的「移除」只需讓 `head` 前進，不需要實際搬移陣列元素，所以單次過期是 O(1)。
  - **空間**：只移動指標會留下後遺症——被跳過的前端元素永遠賴在物理陣列裡。在嚴格遞減的輸入下（沒有任何尾端淘汰，所有元素都靠 head 過期），物理陣列會一路長到 O(n)，使「結構 O(k)」的承諾跳票。因此加入壓縮：**當 `head ≥ deque.length / 2`（等價於「垃圾數 ≥ 有效數」）時，`slice(head)` 切掉前段並把 `head` 歸零**。由於每次壓縮搬移的有效元素數 ≤ 被跳過的垃圾數，而每個垃圾都來自一次 O(1) 的 `head++`，壓縮成本可被攤銷到這些 `head++` 上，整體仍是 O(1) 攤銷；同時物理陣列長度被壓在 ≤ 2k。

```javascript
popExpired(leftBound) {
  // 移動 head 指標跳過所有過期索引（不搬移陣列）→ 時間 O(1) 攤銷
  while (this.head < this.deque.length && this.deque[this.head] < leftBound) {
    this.head++;
  }
  // 空間回收：head*2 >= length 等價於「垃圾(head) >= 有效(length - head)」
  // 此時壓縮，使物理陣列長度維持 O(k)；搬移量 ≤ 垃圾量，攤銷 O(1)
  if (this.head > 0 && this.head * 2 >= this.deque.length) {
    this.deque = this.deque.slice(this.head);
    this.head = 0;
  }
}
```

**視覺化說明**：

```plaintext
基礎版本（shift）：                 優化版本（head pointer）：
deque: [2, 5, 7]                    deque: [2, 5, 7],  head = 0
popExpired(4):                      popExpired(4):
  shift → [5, 7]  (搬移 2 個元素)    head++ → head = 1  (只移動指標)
  front = 5                           deque[head] = 5    ✓ 同樣結果

定期壓縮觸發（head 過半時回收前端垃圾）：
deque: [2, 3, 5, 7],  head = 2      ← 前 2 格(索引2,3)已過期，是垃圾；有效=[5,7]
  head(2) * 2 = 4 >= length(4)  → 觸發壓縮
  slice(2) → deque: [5, 7],  head = 0
觀察：壓縮只切掉「索引位置」上的垃圾格，deque 內存的原始索引值不變，front 仍正確。
```

**優化後的複雜度**：O(1) 攤銷 / O(n) 最壞（單次呼叫可能連續推進多個過期索引；`head` 指標單向前進，總移動量 ≤ n，壓縮搬移總量亦 ≤ n）。

#### 3. front()

**目的**：取得佇列前端元素的索引（即當前窗口內的極值元素索引）。

**輸入/輸出**：

- 輸入：無
- 輸出：前端元素的索引 `index`；佇列為空時回傳 `undefined`

**不變量**：單調性保證前端永遠是當前候選中的最佳答案。

**基礎實作**（即為最終版本）：

```javascript
front() {
  // Edge case: 空佇列回傳 undefined，由呼叫端處理
  if (this.isEmpty()) return undefined;
  return this.deque[this.head];
}
```

**無優化空間**：單調性保證前端就是極值，查詢只需一次直接索引存取，不需要任何遍歷，本身已是 O(1) 最優。

**複雜度**：O(1)。

---

#### 4. frontValue()

**目的**：取得佇列前端索引對應的「值」（即當前窗口極值本身），便於直接寫入答案。

**輸入/輸出**：

- 輸入：無
- 輸出：`nums[front()]`；佇列為空時回傳 `undefined`

**不變量**：與 `front()` 相同，單調性保證回傳的是當前窗口極值。

**基礎實作**（即為最終版本）：

```javascript
frontValue() {
  const idx = this.front();
  return idx !== undefined ? this.nums[idx] : undefined;
}
```

**無優化空間**：僅在 `front()` 之上做一次取值，無迴圈、無額外資料結構，本身已是 O(1) 最優。提供此方法是為了避免呼叫端誤把 `front()` 回傳的「索引」當成「值」使用。

**複雜度**：O(1)。

---

#### 5. isEmpty()

**目的**：判斷佇列是否為空（無有效候選者）。

**輸入/輸出**：

- 輸入：無
- 輸出：`boolean`

```javascript
isEmpty() {
  return this.head >= this.deque.length;
}
```

**無優化空間**：純指標比較，一步得出，本身已是 O(1) 最優。

**複雜度**：O(1)。

---

#### 6. size()

**目的**：回傳佇列中有效元素的數量。

**輸入/輸出**：

- 輸入：無
- 輸出：`number`

```javascript
size() {
  return this.deque.length - this.head;
}
```

**無優化空間**：純指標運算，一步得出，本身已是 O(1) 最優。

**複雜度**：O(1)。

### 核心操作複雜度總結

| 操作                  | 時間複雜度                | 空間複雜度 | 備註                                                   |
| --------------------- | ------------------------- | ---------- | ------------------------------------------------------ |
| **維護操作**          |                           |            |                                                        |
| push(index)           | **O(1) 攤銷** / O(n) 最壞 | O(1)       | 最壞：尾端全部淘汰；攤銷：每元素最多入隊出隊各一次     |
| popExpired(leftBound) | **O(1) 攤銷** / O(n) 最壞 | O(1) 攤銷  | head 指標單向前進；垃圾 ≥ 有效時壓縮回收，搬移總量 ≤ n |
| **查詢操作**          |                           |            |                                                        |
| front()               | O(1)                      | O(1)       | 單調性保證前端即為極值索引                             |
| frontValue()          | O(1)                      | O(1)       | `nums[front()]`，回傳極值本身                          |
| isEmpty()             | O(1)                      | O(1)       | 指標比較                                               |
| size()                | O(1)                      | O(1)       | 指標運算                                               |

**符號定義**：

- `n`：輸入陣列的元素總數。
- `k`：窗口大小（佇列中最多同時存在的有效索引數）。
- **攤銷（Amortized）**：單次操作最壞成本可能較高，但整段操作的總成本有上界，平均到每次操作後的成本。

**攤銷分析的核心論證**：每個元素在整個滑動窗口過程中，最多被 `push` 進佇列一次、被 `pop`（尾端淘汰）或被 `head` 跳過（前端過期）一次。因此 n 個元素的總維護操作次數上界為 2n；前端壓縮的搬移總量亦 ≤ n（每次搬移量不超過此前累積的垃圾量，而垃圾量等於 `head++` 次數），故每次操作的攤銷成本為 O(1)。

**結構本身的空間複雜度**：O(k)。`head` 指標搭配「垃圾 ≥ 有效時壓縮」確保物理陣列長度維持 ≤ 2k，不會隨總輸入量 n 無限增長。

## ⭐ 抽象化翻譯器

### 識別核心抽象

> 這個工具的核心對象是：**一組候選元素的索引**
>
> 它管理的是這些對象之間的什麼關係？**值的單調順序 + 時間的新舊順序（索引大小）**

Monotonic Queue 的本質是同時利用兩個維度進行淘汰：

1. **值維度**：新元素「支配」舊元素（更大或更小）→ 尾端淘汰。
2. **時間維度**：舊元素超出窗口範圍 → 前端過期移除。

### 建立映射維度

| 維度             | 要回答的問題                                   | 這個答案決定了什麼                       |
| ---------------- | ---------------------------------------------- | ---------------------------------------- |
| 維度 1：候選集合 | 問題中的「什麼」是候選元素？                   | `nums` 陣列的定義、佇列存什麼            |
| 維度 2：支配關係 | 什麼條件下新元素「淘汰」舊元素？               | `shouldEvict` 函式的方向（`<=` 或 `>=`） |
| 維度 3：淘汰時機 | 候選什麼時候「過期」？窗口邊界如何移動？       | `popExpired` 的 `leftBound` 計算邏輯     |
| 維度 4：答案提取 | 從佇列前端取出的值，如何轉換為題目要求的答案？ | `frontValue()` 回傳極值後的後處理邏輯    |

### 映射範例

**題目 A**：[239. Sliding Window Maximum](https://leetcode.com/problems/sliding-window-maximum/)

| 維度   | 具體問題                               | 抽象映射                                                                       |
| ------ | -------------------------------------- | ------------------------------------------------------------------------------ |
| 維度 1 | nums 陣列中的每個元素                  | 候選元素 = nums 的索引                                                         |
| 維度 2 | 找最大值 → 較大的值支配較小的          | `shouldEvict = (backVal, newVal) => backVal <= newVal`（Monotonic Decreasing） |
| 維度 3 | 窗口大小固定為 k，左邊界 = `i - k + 1` | `popExpired(i - k + 1)`                                                        |
| 維度 4 | 每個窗口的最大值                       | `result.push(mq.frontValue())`                                                 |

**從映射到實作的關鍵步驟**：

```javascript
for (let i = 0; i < nums.length; i++) {
  mq.push(i); // 維度 1 + 2：加入候選，自動淘汰被支配者
  mq.popExpired(i - k + 1); // 維度 3：移除過期候選
  if (i >= k - 1) {
    result.push(mq.frontValue()); // 維度 4：提取答案（直接取值，避免誤用索引）
  }
}
```

**題目 B**：[1438. Longest Continuous Subarray With Absolute Diff Less Than or Equal to Limit](https://leetcode.com/problems/longest-continuous-subarray-with-absolute-diff-less-than-or-equal-to-limit/)

| 維度   | 具體問題                                          | 抽象映射                                  |
| ------ | ------------------------------------------------- | ----------------------------------------- |
| 維度 1 | nums 陣列中的每個元素                             | 候選元素 = nums 的索引                    |
| 維度 2 | 同時需要最大值和最小值 → **兩個** Monotonic Queue | maxQueue（遞減）+ minQueue（遞增）        |
| 維度 3 | 窗口大小不固定；當 `max - min > limit` 時縮小窗口 | 動態計算 leftBound                        |
| 維度 4 | 窗口長度 `i - left + 1` 的最大值                  | `result = Math.max(result, i - left + 1)` |

**從映射到實作的關鍵步驟**：維度 2 需要兩個佇列協同工作，維度 3 不再是固定 `k` 而是根據條件動態收縮。

## 🔍 觸發器（模式識別）

### 正向觸發器

看到什麼特徵應該想到 Monotonic Queue？

| 層級     | 特徵                                                           | 為什麼這個特徵指向 Monotonic Queue             |
| -------- | -------------------------------------------------------------- | ---------------------------------------------- |
| 關鍵字   | 「滑動窗口的最大/最小值」                                      | 這是 Monotonic Queue 的經典定義場景            |
| 關鍵字   | 「子陣列」+「極值」+「固定/可變長度」                          | 子陣列 = 連續窗口，極值 = 需要單調性維護       |
| 結構     | 輸入是一維陣列，需要在連續區間上反覆查詢極值                   | 反覆查詢 = 需要 O(1) 查詢，連續區間 = 滑動窗口 |
| 目標     | 要求 O(n) 或 O(n log n) 解，暴力法是 O(n × k)                  | O(n) 要求排除了每次窗口重新掃描的可能性        |
| 操作模式 | 問題可以分解為「加入新元素 + 移除最舊元素 + 查詢極值」三步迴圈 | 這正是 Monotonic Queue 的三個核心操作          |

**一句話觸發規則**：**當你需要在「滑動的連續區間」上反覆查詢「極值」時，想到 Monotonic Queue。**

### 反向觸發器

什麼情況下「看起來適用但其實不是」？

| 陷阱情境                                   | 為什麼不適用                                               | 更好的選擇                                     |
| ------------------------------------------ | ---------------------------------------------------------- | ---------------------------------------------- |
| 需要「動態插入 + 刪除任意位置 + 查詢極值」 | Monotonic Queue 只能前端移除、尾端插入，不支援任意位置操作 | Balanced BST（如 AVL Tree）或 Sorted Set       |
| 需要「第 K 大/小」而非「最大/最小」        | Monotonic Queue 只維護極值候選，無法快速取得第 K 個        | Heap（Priority Queue）或 Order Statistics Tree |
| 窗口不是連續的（如每隔 2 個元素取一個）    | 「過期」判斷基於連續索引遞增，非連續窗口的過期邏輯不匹配   | 根據具體問題選擇 Heap 或 Segment Tree          |
| 需要同時支援「增加元素」和「修改已有元素」 | 修改已有元素會破壞已建立的淘汰關係，無法高效修復           | Segment Tree 或 Sparse Table                   |

### 與類似工具的決策點

```plaintext
連續區間上反覆查詢極值
├── 窗口大小固定 + 只需最大/最小 → Monotonic Queue（O(n) 最優）
├── 窗口大小可變 + 只需最大/最小 → Monotonic Queue + Two Pointers
├── 需要第 K 大/小 → Heap / Priority Queue（O(n log k)）
├── 區間任意查詢（不一定連續滑動）→ Segment Tree / Sparse Table（O(n log n) 建構 + O(1) 或 O(log n) 查詢）
└── 需要動態插入/刪除 + 區間查詢 → Balanced BST / Segment Tree with Lazy Propagation
```

## ⚠️ 注意事項

### 常見陷阱

| 陷阱                         | 錯誤寫法                             | 正確寫法                            | 為什麼                                                                                                                                  |
| ---------------------------- | ------------------------------------ | ----------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| 嚴格 vs 非嚴格比較           | `nums[back] < nums[i]`               | `nums[back] <= nums[i]`（求最大時） | 用嚴格 `<` 時，遇到相同值不會淘汰舊的，導致較舊的相同值留下；它較早過期，過期後可能少了候選。用 `<=` 保留較新的相同值，存活更久，較安全 |
| 過期檢查的邊界（off-by-one） | `this.deque[this.head] <= leftBound` | `this.deque[this.head] < leftBound` | `leftBound` 是窗口左邊界（inclusive），等於 `leftBound` 的索引仍在窗口內。用 `<=` 會把恰在邊界上的有效元素也誤刪                        |
| 誤把 front() 當成「值」      | `result.push(mq.front())`            | `result.push(mq.frontValue())`      | `front()` 回傳的是「索引」不是「值」。直接 push 會把索引當答案，須再查 `nums[idx]`，或改用 `frontValue()`                               |
| 使用 `Array.shift()`         | `this.deque.shift()`                 | 使用 head 指標 `this.head++`        | `shift()` 是 O(n) 操作，會破壞整體 O(n) 的攤銷保證                                                                                      |
| 忘記窗口尚未填滿時不取答案   | 從 `i = 0` 就開始收集結果            | `i >= k - 1` 時才收集               | 前 `k - 1` 個元素窗口尚未填滿，取出的極值不代表完整窗口                                                                                 |

### 常見變體

| 變體                       | 修改內容                                                                                | 適用場景                                      |
| -------------------------- | --------------------------------------------------------------------------------------- | --------------------------------------------- |
| Monotonic Decreasing Queue | `shouldEvict = (backVal, newVal) => backVal <= newVal`                                  | 求窗口最大值（Sliding Window Maximum）        |
| Monotonic Increasing Queue | `shouldEvict = (backVal, newVal) => backVal >= newVal`                                  | 求窗口最小值（Sliding Window Minimum）        |
| 雙 Monotonic Queue         | 同時維護一個 max queue 和一個 min queue                                                 | 需要同時追蹤窗口的最大和最小值（如 LC 1438）  |
| 存值而非存索引             | deque 直接存值，`popExpired` 改為比較值相等性（失去以索引判斷過期的能力，僅限固定窗口） | 窗口大小固定且不需要索引資訊的簡化場景        |
| 搭配 DP                    | Monotonic Queue 優化 DP 轉移中的 `max/min(dp[j])` 查詢（`j` 在某個滑動範圍內）          | DP 轉移式包含區間極值查詢，如 LC 1425, LC 375 |
