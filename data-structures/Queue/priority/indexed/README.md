# Indexed Priority Queue

## 🎯 設計理念

### 存在意義

**這個工具被發明出來，是為了解決什麼類型的問題？**

普通 Priority Queue 能 O(log n) 取極值，卻無法 O(log n)「修改某個既存元素的優先級」——因為它根本不知道那個元素現在躺在 heap 的哪個位置，要找位置得 O(n) 線性掃描。但 Dijkstra、Prim 這類演算法的核心動作正是「鬆弛時把某個既存節點的 key 改小」（decreaseKey）。

- **在它出現之前的痛點**：沒有定位能力時，常見替代法是 **lazy deletion**——把同一節點的新值再 push 一次，出列時用 `dist` 表判斷並丟棄過期項（stale）。代價是 heap 元素數從 O(V) 膨脹到 O(E)，且每次出列都要額外做 stale 檢查。
- **核心價值主張**：替每個元素配一個穩定 index，並維護「index → heap 位置」的反向映射，達成 **O(1) 定位 + O(log n) 降鍵**，讓 heap 規模穩定維持在 O(V)。
- **一句話本質**：一個「可以指名道姓修改某個既存元素優先級」的 priority queue。

### 心智模型

**用生活化的類比建立直覺**

類比：**醫院候診的「分診台 + 病歷號對照表」**

- 普通 PQ ＝ 只認「最危急者先看」的分診台。要把「3 號病人」的危急度調高，得翻遍整疊病歷才找得到他在隊伍何處。
- IPQ 多做兩件事：給每位病人一個固定**病歷號**（index），櫃台另備一本**對照表**（`qp`：病歷號 → 目前排第幾）。要調整某人優先級時，查對照表 O(1) 就知道他在哪，再就地修復隊伍順序。

局限：對照表大小固定（`maxN`），病歷號必須是 `[0, maxN)` 的整數。若 id 稀疏、非整數或上界未知，這個陣列式對照表就不適用（需改用 Map 變體，見「常見變體」）。

## 🏗️ 抽象結構

### 核心組件

| 組件         | 功能                         | 為什麼需要                                                     |
| ------------ | ---------------------------- | -------------------------------------------------------------- |
| `keys[i]`    | index `i` → 其優先級 key     | 修改、查詢 key 的依據；heap 比較時透過它取值                   |
| `pq[k]`      | heap 位置 `k` → 存放的 index | heap 真正排序的對象是 **index**（不是 key）                    |
| `qp[i]`      | index `i` → 目前的 heap 位置 | O(1) 定位的關鍵；`-1` 表示不在隊列中，亦作為 `contains` 的判據 |
| `n`          | 當前元素數量                 | 決定 heap 有效範圍 `[0, n)`，是 `sink` 邊界與末端位置的依據    |
| `comparator` | 決定誰優先級較高             | 把「min / max」抽象成比較方向，一份程式碼支援兩種              |

### 狀態表示

- **底層資料結構：三個平行陣列（`keys` / `pq` / `qp`），heap 隱式存在 `pq` 的 `[0, n)` 區段。**
  - 判準：本工具需要「依鍵隨機定位某個既存元素」→ 必須有 `index → 位置` 的 O(1) 映射。因為 index 限定為 `[0, maxN)` 的連續整數，用**陣列**即可（常數比 HashMap 更低）；若 index 稀疏才退而求其次用 Map。
  - 為何不只用一個 heap：heap 本身能取極值，但不具「反向定位」能力。要支援 O(1) 定位，就需在 heap 之外再維護一個 `index → 位置` 的反向映射 `qp`；而這個映射必須在**每次節點交換的當下同步更新**，屬於 heap 操作的內部不變量，無法靠一個「只會交換值、不回報位置變動」的結構從外部疊加。（基於此特性而做的依賴選擇與既有實作比較，見「最後 → 依賴決策」，此處不混入工程紀律。）
- **初始狀態**：`n = 0`；`keys` 全為 `undefined`；`pq`、`qp` 全為 `-1`。
- **合法狀態規則（可執行）**：一個狀態合法，當且僅當同時滿足
  1. 對每個在隊列中的 index `i`：`pq[qp[i]] === i`；
  2. 對每個 heap 位置 `k ∈ [0, n)`：`qp[pq[k]] === k`；
  3. `pq[0..n-1]` 滿足 heap 性質：依 `comparator`，每個節點優先級不低於其子節點。
     照這三條逐一檢查，即可判定任意狀態是否合法。

### 核心操作

1. **contains / keyOf（定位）— IPQ 的根本優化**

**目的**：判斷某 index 是否在隊列中、取得其 key，並為 `decreaseKey`/`delete` 提供 O(1) 定位。

**輸入/輸出**：輸入 index `i`；輸出布林或 key。

**不變量**：不修改任何結構，純查詢。

**基礎實作**：

```javascript
// 沒有反向映射時，只能掃描整個 heap 找 index
contains(i) {
  for (let k = 0; k < this._n; k++) if (this._pq[k] === i) return true;
  return false;
}
```

**基礎版本的問題**：

- 最壞情況 O(n)。在 Dijkstra 中，每次鬆弛都要先定位節點，會把整體拉到 O(V·E) 級。

**優化策略**：反向映射 `qp`

- 核心思想：用空間換時間，維護 `index → heap 位置` 的對照表。
- 為什麼能解決問題：判斷與定位變成單次陣列存取。代價是每次 `swap` 都要多回寫兩筆 `qp`（仍是 O(1)）。

```javascript
contains(i) { return this._qp[i] !== -1; } // O(1)
```

**視覺化說明**：此操作為單次陣列存取（讀 `qp[i]` 或 `keys[i]`），不改變任何結構、無節點移動，故無對應的結構變化圖示。

**優化後的複雜度**：O(1)。

---

2. **isEmpty / size / minIndex / minKey（O(1) 存取器）— 無優化空間**

**目的**：回報隊列空否（`isEmpty`）、元素數（`size`）、極值的 index（`minIndex`）與極值的 key（`minKey`）。

**輸入/輸出**：無輸入；輸出布林 / 數字 / index / key。

**不變量**：純查詢，不修改結構。

**基礎實作**：

```javascript
isEmpty() { return this._n === 0; }
size()    { return this._n; }
minIndex(){ return this._pq[0]; }            // heap 根節點存的 index
minKey()  { return this._keys[this._pq[0]]; } // 再經 keys 取其 key
```

**無優化空間的理由**：四者皆為單次欄位 / 陣列存取——`n` 直接記錄數量，heap 性質保證極值恆在根 `pq[0]`，故取極值無需搜尋。已是 O(1)，沒有更低的下界可優化。（`minIndex`/`minKey` 在空隊列拋 `"Priority queue underflow"`，見實作。）

**視覺化說明**：均為 O(1) 直接存取，無結構變化，無需圖示。

**複雜度**：O(1)。

---

3. **insert（插入）**

**目的**：把 index `i` 與其 key 加入隊列。

**輸入/輸出**：輸入 `(i, key)`；無回傳。

**不變量**：插入後 `pq[qp[i]] === i` 成立，且全堆滿足 heap 性質。

**基礎實作**：

```javascript
// 放到末端建立三向映射，再上浮修復
this._qp[i] = this._n;
this._pq[this._n] = i;
this._keys[i] = key;
this._swim(this._n); // 注意：先 swim，再 this._n++
this._n++;
```

**基礎版本的問題**：易錯點在順序——若先 `this._n++` 再 `swim(this._n)`，`swim` 的起點位置就錯了一格。

**優化策略**：無額外優化，標準 heap 插入即 O(log n)；重點是 `swim` 過程中每次交換都靠 `_exch` 同步更新 `qp`。

**視覺化說明**（Min IPQ，插入 `idx=3, key=5`；節點標記為 `index/key`）：

```plaintext
插入前（n=2）           放到末端 pos2          swim：pos2 與父 pos0 比較
   pos0: 1/20            pos0: 1/20            5 < 20 → 交換
      |                  /        \                pos0: 3/5
   pos1: 0/50      pos1: 0/50   pos2: 3/5  ──▶    /        \
                                              pos1: 0/50  pos2: 1/20
qp = {0→1, 1→0}     qp = {0→1,1→0,3→2}        qp = {0→1, 1→2, 3→0}
                                              （交換 pos0↔pos2 時，qp[3]、qp[1] 同步更新）
```

**優化後的複雜度**：O(log n)。

---

4. **decreaseKey（降鍵）— 招牌操作**

**目的**：把既存 index `i` 的 key 改成「優先級更高」的新值。

**輸入/輸出**：輸入 `(i, key)`，新 key 必須嚴格更優先；無回傳。

**不變量**：修改後 heap 性質恢復；映射不變。

**基礎實作（沒有 IPQ 時的 lazy deletion 替代法）**：

```javascript
// 不修改舊項，直接把新值再 push 一次；出列時用外部 dist 表丟棄過期項
pq.push([newDist, node]);
// dequeue 時：if (d > dist[node]) continue; // stale，跳過
```

**基礎版本的問題**：

- heap 元素數從 O(V) 膨脹到 O(E)；每次出列都要做 stale 檢查；空間 O(E)。

**優化策略**：O(1) 定位 + 單向修復

- 核心思想：直接改 `keys[i]`，再從 `qp[i]` 上浮。
- 為什麼能解決問題：方向已知（變得更優先 → 只可能往上），故只需 `swim`，且 heap 維持 O(V)。

```javascript
decreaseKey(i, key) {
  this._keys[i] = key;
  this._swim(this._qp[i]); // 更優先 → 只可能往上
}
```

**視覺化說明**（兩張圖分別獨立；節點標記 `index/key`）：

```plaintext
〔圖 A〕IPQ：原地降鍵 + 上浮（元素數恆為 V）
   降鍵前                              decreaseKey(idx=0, 100→10) 後
       pos0: 1/50                          pos0: 0/10   ← idx0 上浮到根
      /        \           swim          /        \
 pos1: 0/100  pos2: 2/75   ─────▶   pos1: 1/50  pos2: 2/75
 ↑ idx0 的 key 100 改成 10           （idx0 從 pos1 上浮到 pos0）
```

```plaintext
〔圖 B〕lazy deletion：重複插入致 heap 膨脹（元素數可達 E）
 同一個 idx0 被以新 key 再插入一次，舊項仍留在 heap：
   [ 0/10(新) ] [ 1/50 ] [ 2/75 ] [ 0/100(舊, stale) ] ...
 出列時：先吐 0/10；日後遇到 0/100 因 dist[0]=10 < 100 → 判定 stale 丟棄
```

**優化後的複雜度**：O(log n)，空間 O(1)。

---

5. **increaseKey（升鍵）— decreaseKey 的對稱操作**

**目的**：把既存 index `i` 的 key 改成「優先級更低」的新值（與 `decreaseKey` 完全對稱）。

**輸入/輸出**：輸入 `(i, key)`，新 key 必須嚴格更不優先；無回傳。

**不變量**：修改後 heap 性質恢復；映射不變。

**基礎實作 / 問題**：與 `decreaseKey` 同源——沒有 IPQ 時同樣只能 lazy deletion 或 O(n) 重建，痛點相同。

**優化策略**：O(1) 定位 + 單向修復（方向相反）

- 核心思想：改 `keys[i]` 後從 `qp[i]` **下沉**。
- 為什麼是 `sink` 而非 `swim`：優先級變低，只可能被子節點超越往下走，不可能上浮。

```javascript
increaseKey(i, key) {
  this._keys[i] = key;
  this._sink(this._qp[i]); // 更不優先 → 只可能往下
}
```

**視覺化說明**（與 decreaseKey 圖 A 鏡像；節點標記 `index/key`）：

```plaintext
   升鍵前                              increaseKey(idx=0, 5→80) 後
       pos0: 0/5                            pos0: 1/20   ← 較優先的子節點補上
      /        \           sink           /        \
 pos1: 1/20   pos2: 2/75   ─────▶   pos1: 0/80  pos2: 2/75
 ↑ idx0 的 key 5 改成 80              （idx0 從 pos0 下沉到 pos1）
```

**優化後的複雜度**：O(log n)，空間 O(1)。

---

6. **changeKey（改鍵，方向未知）**

**目的**：把既存 index `i` 的 key 改成任意新值（不預設變高或變低）。

**輸入/輸出**：輸入 `(i, key)`；無回傳。

**不變量**：修改後 heap 性質恢復；映射不變。

**基礎版本的問題**：呼叫端若不確定方向，硬套 `decreaseKey`/`increaseKey` 會在方向不符時拋錯或漏修復。

**優化策略**：swim 與 sink 都做

- 核心思想：先 `swim` 再 `sink`，因為任一節點最多只可能往單一方向移動，另一方向會立即停止。
- 為什麼仍是 O(log n)：錯誤方向的那次呼叫在第一步比較就 `break`（O(1)），正確方向最多走樹高一次。

```javascript
changeKey(i, key) {
  this._keys[i] = key;
  this._swim(this._qp[i]); // 變優先則往上
  this._sink(this._qp[i]); // 否則往下；兩者只會發生其一
}
```

**視覺化說明**：實際位移即「圖 A（上浮）」或「increaseKey 圖（下沉）」其一，視新 key 方向而定；`changeKey` 不另引入新機制，故不重複繪圖。

**優化後的複雜度**：O(log n)。

---

7. **delMin / delete（刪極值 / 刪指定）**

**目的**：移除並回傳優先級最高者（`delMin`）；或移除指定 index（`delete`）。

**輸入/輸出**：`delMin` 回傳 index；`delete(i)` 輸入 index、無回傳。

**不變量**：移除後 heap 性質恢復，且被移除 index 的 `qp`、`keys`、原末端 `pq` 皆清乾淨。

**基礎實作**（兩者共用「末端補位 → 修復 → 清映射」骨架，差別只在修復方向）：

```javascript
delMin() {                      // 刪的一定是根
  const min = this._pq[0];
  this._exch(0, this._n - 1);   // 末端換到根
  this._n--;
  this._sink(0);                // 根無父節點 → 只可能往下沉
  this._qp[min] = -1; this._keys[min] = undefined; this._pq[this._n] = -1; // 清映射
  return min;
}

delete(i) {                     // 刪的是任意位置
  const k = this._qp[i];        // O(1) 定位待刪位置
  this._exch(k, this._n - 1);   // 末端換到 k
  this._n--;
  this._swim(k);                // 末端元素可能比 k 的父更優先 → 需上浮
  this._sink(k);                // 也可能比子節點更不優先 → 需下沉（兩者只會發生其一）
  this._keys[i] = undefined; this._qp[i] = -1; this._pq[this._n] = -1; // 清映射
}
```

**基礎版本的問題**：易錯點在「修復方向」。

- `delMin` 刪的永遠是根，補上來的末端元素只可能往下沉，單做 `sink` 就夠；

- `delete(i)` 刪的是任意位置，補上來的末端元素相對新位置可能太優先（要上浮）也可能太不優先（要下沉），**只做其一會在特定資料下破壞 heap 性質**。

另一個易錯點是漏清映射：被移除的 index 若沒把 `qp[i]` 設回 `-1`，`contains` 會誤判它仍在隊列中（幽靈 index）。

**優化策略**：`delete` 一律先 `swim` 再 `sink`，兩者最多只有一個會實際移動元素（錯誤方向在第一次比較就 `break`，仍是 O(log n)）；並務必清映射。

**視覺化說明 1**（`delMin`：刪根 → 只下沉；節點標記 `index/key`）：

```plaintext
   刪除前(n=3)            末端換到根 + n--          sink 修復
      pos0: 0/10            pos0: 2/75            75 > 子節點 50 → 與較優子節點交換
     /        \            /                          pos0: 1/50
 pos1: 1/50  pos2: 2/75  pos1: 1/50  ──────▶        /
 (回傳 idx0)            (idx2 補到根, n=2)         pos1: 2/75
 清映射：qp[0]=-1, keys[0]=undefined, pq[2]=-1     qp 同步：{1→0, 2→1}
```

**視覺化說明 2**（`delete`：刪非根 → 可能需上浮，這是 `delMin` 永不發生的情形；節點標記 `index/key`）：

```plaintext
delete(idx3)：末端 idx5/8 補到 pos3 後，比其父 pos1/40 更優先 → 觸發 swim（根無父，故 delMin 永不上浮）

   刪除前(n=6)               末端補到 pos3 + n--           swim：8 < 父 40 → 上浮
        0/1                       0/1                          0/1
       /    \                    /    \                       /    \
    1/40    2/2              1/40    2/2                   5/8     2/2
    /  \    /                /  \                          /  \
 3/45 4/50 5/8           5/8  4/50                     1/40 4/50
 (刪 idx3 @ pos3)       (idx5 補到 pos3, n=5)        （idx5 由 pos3 上浮到 pos1）
```

**優化後的複雜度**：O(log n)。

### 核心操作複雜度總結

| 操作                  | 時間複雜度 | 空間複雜度 | 備註                                         |
| --------------------- | ---------- | ---------- | -------------------------------------------- |
| **查詢**              |            |            |                                              |
| isEmpty() / size()    | O(1)       | O(1)       | 直接讀 `n`                                   |
| contains(i)           | O(1)       | O(1)       | 靠反向映射 `qp[i] !== -1`，免 O(n) 掃描      |
| keyOf(i)              | O(1)       | O(1)       | 不在隊列拋錯                                 |
| minIndex() / minKey() | O(1)       | O(1)       | heap 根節點 `pq[0]`；空隊列拋錯              |
| **插入**              |            |            |                                              |
| insert(i, key)        | O(log n)   | O(1)       | 末端放置 + swim；重複 index 拋錯             |
| **修改 key**          |            |            |                                              |
| decreaseKey(i, key)   | O(log n)   | O(1)       | **招牌操作**；只需 swim，方向錯拋錯          |
| increaseKey(i, key)   | O(log n)   | O(1)       | 對稱操作；只需 sink，方向錯拋錯              |
| changeKey(i, key)     | O(log n)   | O(1)       | 方向未知 → swim + sink（錯誤方向立即 break） |
| **刪除**              |            |            |                                              |
| delMin()              | O(log n)   | O(1)       | 換末端 + sink + 清映射；空隊列拋錯           |
| delete(i)             | O(log n)   | O(1)       | O(1) 定位後 swim **且** sink + 清映射        |

**符號定義**：

- `n`：當前隊列中的元素數量。
- `log n`：完全二元樹的高度。

**結構本身的空間複雜度**：O(maxN)（`keys`、`pq`、`qp` 三個長度 `maxN` 的陣列）。

## ⭐ 抽象化翻譯器

### 識別核心抽象

> 這個工具的核心對象是：**一群被「穩定 id」標記、且優先級會隨演算法進程反覆變動的元素**。
>
> 它管理的關係是：每個 id ↔ 其當前優先級，並隨時能取出「目前最優先的 id」、也能對「任一指定 id」改變其優先級。

### 建立映射維度

| 維度   | 要回答的問題                                       | 這個答案決定了什麼                     |
| ------ | -------------------------------------------------- | -------------------------------------- |
| 維度 1 | 問題中那個「穩定 id」是什麼？範圍多大？            | `maxN` 與 index 的編碼方式             |
| 維度 2 | 「優先級（key）」是什麼？比較方向是 min 還是 max？ | `comparator` 的選擇                    |
| 維度 3 | 什麼**事件**會觸發「對既存元素改鍵」？             | `decreaseKey` / `changeKey` 的呼叫時機 |

維度順序的邏輯：先定 id（對象）→ 再定 key（表示）→ 最後定觸發事件（操作）。三者填完，主迴圈的骨架就浮現了。

### 實戰檢查表

```plaintext
題目：_______________

維度 1（穩定 id 是什麼、上界多少）：_______________
維度 2（key 是什麼、min 或 max）：_______________
維度 3（什麼事件觸發改鍵）：_______________

填完後，主迴圈骨架：
  while (!ipq.isEmpty()) {
    const u = ipq.delMin();          // 取目前最優先的 id
    for (鄰居/相關元素 v of u) {
      if (改善條件成立) {             // ← 維度 3 的觸發事件
        if (ipq.contains(v)) ipq.decreaseKey(v, 新key);
        else ipq.insert(v, 新key);
      }
    }
  }
```

### 映射範例

**題目 A**：Dijkstra 單源最短路徑

| 維度   | 具體問題                             | 抽象映射                                               |
| ------ | ------------------------------------ | ------------------------------------------------------ |
| 維度 1 | 穩定 id 是節點編號                   | index = 節點編號，`maxN` = 節點數 V                    |
| 維度 2 | key 是「源點到該節點的最短估計距離」 | min IPQ（距離越小越優先），預設 comparator             |
| 維度 3 | 鬆弛成功（找到更短路徑）時改鍵       | `if (dist[v] > dist[u] + w) decreaseKey(v, dist[u]+w)` |

**從映射到實作的關鍵步驟**：起點 `insert(s, 0)`；每次 `delMin` 取出當前最近節點 `u`，對其每條出邊嘗試鬆弛，成功則 `contains` 判斷該用 `decreaseKey` 還是 `insert`。相較 lazy deletion，heap 元素數恆為 O(V)、無需 stale 檢查。

**題目 B**：Prim 最小生成樹（與 A 底層同機制，介面一致）

| 維度   | 具體問題                           | 抽象映射                                      |
| ------ | ---------------------------------- | --------------------------------------------- |
| 維度 1 | 穩定 id 是節點編號                 | index = 節點編號，`maxN` = V                  |
| 維度 2 | key 是「連到已選樹的最小單邊權重」 | min IPQ                                       |
| 維度 3 | 某邊提供更小的接入成本時改鍵       | `if (w(u,v) < key[v]) decreaseKey(v, w(u,v))` |

> A、B 底層是同一套機制（取極值 → 鬆弛鄰居 → 降鍵），故介面寫法一致。**真正的分流維度是「key 的語意」**：Dijkstra 的 key 是「累積路徑長」，Prim 的 key 是「單一接入邊的權重」——這是題目本質差異，而非寫法差異。

## 🔍 觸發器(模式識別)

### 正向觸發器

| 層級     | 特徵                                                     | 為什麼這個特徵指向這個工具             |
| -------- | -------------------------------------------------------- | -------------------------------------- |
| 關鍵字   | 「最短路徑」「最小生成樹」「動態調整某元素優先級」       | 這些演算法的核心動作就是對既存元素降鍵 |
| 結構     | 每個元素有**穩定身分**，且優先級會在過程中**被多次更新** | 穩定 id + 可變 key 正是 IPQ 的對象模型 |
| 目標     | 反覆取極值 **且** 反覆對「指定既存元素」改鍵             | 普通 heap 只滿足前者                   |
| 操作模式 | 程式中出現 `decreaseKey` / `update(node, newPriority)`   | 普通 PQ 無此操作                       |

**一句話觸發規則**：當「取極值」與「對既存元素改優先級」**同時且反覆**出現 → 用 IPQ。

### 反向觸發器

| 陷阱情境                     | 為什麼不適用                     | 更好的選擇                 |
| ---------------------------- | -------------------------------- | -------------------------- |
| 只取極值、從不修改既存元素   | 反向映射純屬多餘負擔，是過度設計 | 普通 Heap / Priority Queue |
| id 稀疏 / 非整數 / 上界未知  | 固定長度陣列 `qp` 無法定址       | Map-based IPQ              |
| 只需一次性 top-k（靜態資料） | 不需要動態改鍵                   | 普通 Heap + `delMin` k 次  |

### 與類似工具的決策點

分支依「問題性質」分流，彼此互斥；lazy deletion 是分支 3 之下的**實作取捨**，非並列分支：

```plaintext
[需要反覆取極值的問題]
├── 從不修改既存元素的優先級 → 普通 Heap / PriorityQueue（反向映射純屬多餘）
├── 會修改既存元素，但 id 稀疏 / 非整數 / 上界未知 → Map-based IPQ
└── 會修改既存元素，且 id 為連續整數、上界已知 → 本工具（陣列式 IPQ，常數最低）
        └─（實作取捨）若不願引入 IPQ 的程式碼複雜度，可改用普通 PQ + lazy deletion，
           代價是 heap 規模膨脹至 O(E) 並需 stale 檢查
```

## ⚠️ 注意事項

### 常見陷阱

| 陷阱                         | 錯誤寫法                          | 正確寫法                                         | 為什麼                                    |
| ---------------------------- | --------------------------------- | ------------------------------------------------ | ----------------------------------------- |
| `exch` 只換 `pq` 沒回寫 `qp` | `[pq[i], pq[j]] = [pq[j], pq[i]]` | 換 `pq` 後再 `qp[pq[i]]=i; qp[pq[j]]=j`          | 否則 `pq[qp[i]]===i` 被破壞，後續定位全錯 |
| `insert` 計數與上浮順序顛倒  | 先 `n++` 再 `swim(n)`             | 先 `swim(n)` 再 `n++`                            | `swim` 的起點位置會錯一格                 |
| 把 key 直接放進 heap 排序    | heap 存 key、另存 index           | heap 只存 **index**，比較時才 `keys[pq[k]]` 取值 | 排 index 才能用一份 `qp` 反向定位         |
| `delete(i)` 只 `sink`        | 換末端後僅 `sink(k)`              | `swim(k)` **且** `sink(k)`                       | 換上來的元素可能比原位置更優先，需要上浮  |
| `decreaseKey` 方向用反       | 對較差的新 key 呼叫 `decreaseKey` | 較差用 `increaseKey`，未知用 `changeKey`         | 只 `swim` 無法修復「應下沉」的情況        |

### 常見變體

| 變體          | 修改內容                                   | 適用場景                                            |
| ------------- | ------------------------------------------ | --------------------------------------------------- |
| Max IPQ       | 建構子 comparator 改 `(a, b) => b - a`     | 取最大優先級（如最大頻寬路徑）                      |
| Map-based IPQ | `qp`、`keys` 由陣列改為 `Map`              | id 稀疏、非整數、或上界未知                         |
| d-ary IPQ     | `_left/_right/_parent` 改為 d 元樹索引公式 | decreaseKey 密集時，降低樹高加速上浮（delMin 略慢） |
| 整合既有 Heap | 在 Heap 的 `swap` 注入「位置變動回呼」     | 想統一 heap 程式碼，但須改寫 Heap 內部不變量        |
