# Monotonic Stack（單調堆疊）

## 🎯 設計理念

### 存在意義

**這個工具被發明出來，是為了解決什麼類型的問題？**

**痛點**：對每個元素，找到它「下一個更大／更小元素」（Next Greater/Smaller Element，NGE/NSE）。沒有專用工具時的替代方案是「對每個元素往右逐一掃描」，這是雙層迴圈，暴力解 O(n²)。當 n ≤ 10⁵ 時，O(n²) 約 10¹⁰ 次運算，無法在時限內完成。

**價值主張**：在遍歷過程中「順便」把永遠不可能成為答案的候選者淘汰掉，使每個元素最多入棧一次、出棧一次，整體用 O(n) 解決所有元素的 NGE/NSE 查詢。

**一句話本質**：一個會在新元素到來時「自動淘汰過時候選者」的堆疊，讓棧頂永遠是最近且最該被比較的候選者，從而在 O(n) 內回答每個元素的「最近更大／更小」問題。

### 心智模型

**用生活化的類比建立直覺**

**類比：排隊看舞台（站立區）**

一排人站著看舞台，身高不同。以「找右邊第一個更高的人」（NGE）為例：

| 操作           | 生活場景                                             | Monotonic Stack 操作          |
| -------------- | ---------------------------------------------------- | ----------------------------- |
| 新人加入       | 一個人走到隊伍尾端                                   | push 新元素                   |
| 被擋住的人離開 | 排在他前面、比他矮的人，反正被新來的高個子擋住了     | 清理（pop）比新元素矮的候選者 |
| 留下的人       | 棧裡只剩「由前到後一路變矮」的人，每個都還沒被誰擋住 | 棧內維持單調遞減              |

**核心直覺**：當一個更高的人走進來，所有排在前面、比他矮、且還沒等到「右邊更高者」的人，答案就此確定——確定是這個新來的高個子（他更高，又出現得更晚，剛好是被擋者右邊最近的更高者）。

**局限性**（類比會誤導的地方）：

- 真實排隊可以側身偷看，但 Monotonic Stack 只處理「嚴格的前後遮擋」關係。
- 它只回答「最近的」更大／更小，不回答「第 2 近」或「右邊所有更大者」。
- 類比是空間上的遮擋，Monotonic Stack 是索引（時間）上的淘汰。

## 🏗️ 抽象結構

### 核心組件

| 組件                   | 功能                                     | 為什麼需要                                                                        |
| ---------------------- | ---------------------------------------- | --------------------------------------------------------------------------------- |
| **底層 Stack（堆疊）** | 存放尚未確定答案的候選元素（通常存索引） | 沒有 LIFO 就無法保證「最近加入的候選者在最上面」，新元素才能正確找到最近的對象    |
| **單調性不變量**       | 棧內元素由底到頂維持單調（遞增或遞減）   | 維持單調後，棧頂就是「最該被新元素比較／淘汰」的那一個；少了它，清理就失去依據    |
| **shouldPop 比較函式** | 決定新元素到來時，棧頂是否該被淘汰       | 把「找更大／更小」抽象成一個可注入條件，讓四種方向變體共用同一套清理邏輯          |
| **呼叫端結果陣列**     | 記錄每個元素的答案（NGE/NSE/PGE/PSE）    | push 回傳被淘汰的 `popped` 與清理後棧頂 `prevTop`，呼叫端據此回填，職責不混進結構 |

### 狀態表示

**底層資料結構**：Stack（LIFO）。原因：清理、入棧、讀答案這三個動作都只發生在「最近加入」的那一端（top）——新元素從 top 入棧、被淘汰的候選者從 top 彈出、要回填的答案也是讀 top。全程都不需要從另一端（底端）存取或移除元素，因此單端存取的 Stack 就足夠，不必用到雙端的 Deque。

> [!NOTE]
> 這也正是它與 Monotonic Deque 的分界：
>
> - 滑動視窗會讓元素從前端「過期離開」，多出一個操作端，才需要 Deque
> - Monotonic Stack 沒有這個需求，故用單端的 Stack

**Stack 內的值？**

- 存**索引**：索引既能透過 `arr[index]` 取值，又能定位結果陣列的位置。
- 存值本身的情境：只需要值本身、不需回填特定位置時（較少見）。

**初始狀態**：

```plaintext
stack  = 空堆疊（new Stack()）
result = 呼叫端維護的結果陣列，預設值依題意（常見 -1 或 0）
```

**單調性定義規則**：

```plaintext
單調遞增棧（Monotonically Increasing）：底到頂的值由小到大
  → 棧頂是「目前最大的候選者」，新來一個更小的就會觸發清理
  → 用途：找 Next Smaller（NSE）/ Previous Smaller（PSE）

單調遞減棧（Monotonically Decreasing）：底到頂的值由大到小
  → 棧頂是「目前最小的候選者」，新來一個更大的就會觸發清理
  → 用途：找 Next Greater（NGE）/ Previous Greater（PGE）

⚠️ 嚴格 vs 非嚴格：棧到底是「嚴格單調」還是「允許相等並存」，不是固定的，而是由 shouldPop 決定：
   - 內部清理條件判斷用 >  / <  （嚴格不等） → 相等元素會並存   → 非嚴格單調
   - 內部清理條件判斷用 >= / <= （含等號）   → 相等元素會被清掉 → 嚴格單調
   例：arr = [2, 2] 找 NGE 用 (i,t)=>arr[i]>arr[t]，第二個 2 不會清掉第一個 2，兩者並存
       → 此時棧是「非嚴格」遞減。
```

> **命名慣例注意**：本文件統一以「棧底到棧頂的方向」定義：
>
> - Monotonic Increasing Stack = 小到大
> - Monotonic Decreasing Stack = 大到小

### 核心操作

Monotonic Stack 的核心只有一個複合操作——**push（單調入棧）**，它在普通堆疊的 push 前面加上「清理」關卡。`pop / peek / isEmpty / size` 與普通 Stack 完全相同，皆為 O(1) 且無優化空間。

---

1. **push（單調入棧）**

**目的**：把新元素加入棧中，同時維持單調性；並回報「這次清理掉誰」與「清理後棧頂是誰」，讓呼叫端能回填答案。

**輸入／輸出**：

- 輸入：新元素（通常是索引 `i`）。
- 輸出：`{ popped, prevTop }`。`popped` 是本次被淘汰的元素（Next 類問題用）；`prevTop` 是清理後、入棧前的棧頂（Previous 類問題用）。

**不變量**：執行前後，棧內元素的值保持單調（嚴格與否由 `shouldPop` 決定）。

**基礎做法**（不用堆疊時，回答 NGE 查詢的最直覺寫法）：

> 要被優化的單位是「回答每個元素的最近更大／更小查詢」這整件事，push 就是取代下面這段暴力掃描的機制。

```javascript
// 暴力法：對每個元素往右掃描，找第一個更大的
function nextGreaterBrute(arr) {
  const n = arr.length;
  const result = new Array(n).fill(-1);
  for (let i = 0; i < n; i++) {
    for (let j = i + 1; j < n; j++) {
      if (arr[j] > arr[i]) {
        result[i] = arr[j];
        break;
      }
    }
  }
  return result;
}
// 時間 O(n²)，空間 O(n)
```

**基礎版本的問題**：

- 最壞情況：陣列嚴格遞減（如 `[5, 4, 3, 2, 1]`），每個元素都往右掃到底，總比較次數 ≈ n²/2，退化為 O(n²)。
- 為什麼會發生：每次往右掃描都從頭重來，完全沒利用「前一次掃描已經比較過的結果」。

**優化策略：單調清理（monotonic eviction）**

- **核心思想（一句話）**：用棧存「還沒找到答案的候選者」，新元素一次把所有比它弱的候選者淘汰，因為這些候選者既比新元素弱、又比它早出現，未來永遠贏不了。
- **為什麼能解決問題？** 一個候選者一旦被某個新元素淘汰，就永遠出棧、不再被掃描；於是每個元素一生只入棧一次、出棧一次，n 個元素總共最多 2n 次操作，把重複掃描完全消掉。而棧的 LIFO 保證被淘汰者碰到的第一個新元素，就是它「最近的」答案。

```javascript
// 以 MonotonicStack 找每個元素的 Next Greater Element
function nextGreater(arr) {
  const result = new Array(arr.length).fill(-1);
  // 遞減棧：新值 > 棧頂值 時，棧頂找到了它的 NGE（觸發者就是答案）
  const ms = new MonotonicStack((i, topIdx) => arr[i] > arr[topIdx]);
  for (let i = 0; i < arr.length; i++) {
    const { popped } = ms.push(i);
    for (const idx of popped) result[idx] = arr[i]; // Next 類：讀 popped
  }
  return result; // 結束後仍在棧中者 → 右邊無更大，保持 -1
}
```

**視覺化說明**（單看本圖即可理解 push 的清理機制）：

```plaintext
輸入：arr = [2, 1, 5, 3, 4]，找 NGE（遞減棧，存索引；括號為對應值）

i=0  值2  棧[]            →  空棧，直接入棧            →  棧[0(2)]
i=1  值1  棧[0(2)]        →  1 > 2? 否，不清理         →  棧[0(2), 1(1)]
i=2  值5  棧[0(2), 1(1)]  →  5 > 1? 是 → pop 1, result[1]=5
                          →  5 > 2? 是 → pop 0, result[0]=5
                          →  入棧                      →  棧[2(5)]
i=3  值3  棧[2(5)]        →  3 > 5? 否，不清理         →  棧[2(5), 3(3)]
i=4  值4  棧[2(5), 3(3)]  →  4 > 3? 是 → pop 3, result[3]=4
                          →  4 > 5? 否，停止           →  棧[2(5), 4(4)]

結束：索引 2、4 留在棧中 → 右邊沒有更大者
result = [5, 5, -1, 4, -1]
```

**優化後的複雜度**：

- 時間 O(n)：每個元素最多入棧一次、出棧一次，共 ≤ 2n 次操作。
- 空間 O(n)：棧最壞存放全部 n 個元素。

2. **pop / peek / isEmpty / size（基礎堆疊操作）**

**目的**：`pop` 移除棧頂並回傳；`peek` 查看棧頂但不移除；`isEmpty` 判斷棧空；`size` 取得元素數量。在直方圖、環形等需要「逐次掌握彈出當下棧頂」的題型，直接使用這些底層操作手動清理。

**輸入／輸出**：皆無或單一回傳值（被移除／查看的元素、布林、數字）。

**不變量**：`pop` 移除的是棧頂，不影響其餘元素相對順序，故單調性自然維持；`peek/isEmpty/size` 為唯讀，不改變狀態。

**無優化空間**：四者都是對棧頂或長度的直接讀寫，本來就是 O(1)，沒有更快的做法，因此不存在「基礎→問題→優化」的演進；它們繼承自底層 Stack，行為與普通堆疊完全相同。

```javascript
ms.peek(); // 讀棧頂
ms.pop(); // 移除棧頂
ms.isEmpty(); // 是否為空
ms.size(); // 元素數量
```

### 核心操作複雜度總結

| 操作             | 時間複雜度                | 空間複雜度 | 備註                              |
| ---------------- | ------------------------- | ---------- | --------------------------------- |
| **單調維護**     |                           |            |                                   |
| push（單次）     | **O(1) 攤銷** / O(n) 最壞 | O(1)       | 最壞：一次清理掉整個棧            |
| push（n 次總計） | O(n) total                | O(n)       | 每元素一生進出各一次，總操作 ≤ 2n |
| **基礎堆疊操作** |                           |            |                                   |
| pop              | O(1)                      | O(1)       | 同普通 Stack，無優化空間          |
| peek             | O(1)                      | O(1)       | 同普通 Stack，無優化空間          |
| isEmpty          | O(1)                      | O(1)       | 同普通 Stack，無優化空間          |
| size             | O(1)                      | O(1)       | 同普通 Stack，無優化空間          |

**符號定義**：

- `n`：通過此堆疊的元素總數（通常等於輸入陣列長度）。
- **攤銷（amortized）**：單次 push 最壞可能清理多個元素，但 n 個元素全程最多被彈出 n 次、入棧 n 次（共 ≤ 2n 次），平均分攤到每次 push 就是 O(1)——這種「最壞單次貴、但總量有上限」的平均成本就叫攤銷。
- **total**：n 次操作的累計總成本（與「攤銷」不同，total 描述整體而非單次平均）。

**結構本身的空間複雜度**：O(n)（最壞所有元素同時在棧內，如嚴格遞減輸入）。

## ⭐ 抽象化翻譯器

### 識別核心抽象

> 這個工具的核心對象是：**陣列中的元素（以索引代表）**
>
> 它管理的是這些對象之間的什麼關係？**「我右邊／左邊最近的、比我更強的鄰居是誰？」——一種基於大小比較的近鄰關係**

### 建立映射維度

四個維度分別對應到「如何配置 MonotonicStack」：

| 維度                 | 要回答的問題                                 | 這個答案決定了什麼                                                     |
| -------------------- | -------------------------------------------- | ---------------------------------------------------------------------- |
| **維度 1：比較方向** | 找「更大」還是「更小」？                     | `shouldPop` 用 `>`（遞減棧、找 Greater）還是 `<`（遞增棧、找 Smaller） |
| **維度 2：搜尋方向** | 找「右邊」（Next）還是「左邊」（Previous）？ | 讀 push 回傳的 `popped`（Next）還是 `prevTop`（Previous）              |
| **維度 3：存什麼**   | 答案要的是「值」「索引」還是「距離」？       | result 怎麼算（`arr[i]` / `idx` / `i - idx`）                          |
| **維度 4：邊界語意** | 找不到時回傳什麼？是否環形？                 | result 預設值、是否走 2n 圈                                            |

**四種基本組合（皆左→右遍歷）**：

```plaintext
                 Next（讀 popped）                   Previous（讀 prevTop）
Greater          遞減棧；shouldPop: arr[i] >  arr[t]  遞減棧；shouldPop: arr[t] <= arr[i]
（更大）         popped 的答案 = arr[i]              prevTop = 答案
Smaller          遞增棧；shouldPop: arr[i] <  arr[t]  遞增棧；shouldPop: arr[t] >= arr[i]
（更小）         popped 的答案 = arr[i]              prevTop = 答案

注意等號：Next 用嚴格（>、<），相等者不互相觸發；
         Previous 用含等號（<=、>=）清理，確保清理後棧頂「嚴格更大／更小」才當答案。
```

### 實戰檢查表

```plaintext
題目：_______________

維度 1（比較方向）：找更大還是更小？ → _______________
  → 更大用遞減棧（shouldPop 用 >）；更小用遞增棧（shouldPop 用 <）

維度 2（搜尋方向）：找 Next 還是 Previous？ → _______________
  → Next → 讀 push 回傳的 popped，逐一回填答案 = 當前 arr[i]
  → Previous → 讀 push 回傳的 prevTop，prevTop 即為當前元素的答案

維度 3（存什麼）：要值／索引／距離？ → _______________
  → 決定 result 的計算方式（arr[i] / idx / i - idx）

維度 4（邊界語意）：找不到回傳什麼？是否環形？ → _______________
  → 決定 result 預設值、是否走 2n 圈

填完後，核心邏輯應該能直接寫出。
```

### 映射範例

**題目 A**:[LC 739] Daily Temperatures —— 答案 = 觸發彈出的元素本身

> 給定每日溫度,回傳每天要等幾天才會遇到更高溫度。

| 維度   | 具體問題              | 抽象映射                                 |
| ------ | --------------------- | ---------------------------------------- |
| 維度 1 | 找更高溫 → 找「更大」 | 遞減棧,`shouldPop: (i,t)=>arr[i]>arr[t]` |
| 維度 2 | 往右找未來的天 → Next | 讀 `popped`,被彈出者的答案就是當前 `i`   |
| 維度 3 | 「等幾天」= 索引差    | `result[idx] = i - idx`                  |
| 維度 4 | 找不到回傳 0          | result 預設 0                            |

```javascript
function dailyTemperatures(temperatures) {
  const result = new Array(temperatures.length).fill(0); // 維度4：預設 0
  // 維度1：遞減棧
  const ms = new MonotonicStack((i, t) => temperatures[i] > temperatures[t]);
  for (let i = 0; i < temperatures.length; i++) {
    const { popped } = ms.push(i); // 維度2：Next → 讀 popped
    for (const idx of popped) result[idx] = i - idx; // 維度3：索引差，答案 = 觸發者 i
  }
  return result;
}
```

**題目 B**:[LC 84] Largest Rectangle in Histogram —— 答案 = 彈出當下的左右邊界

> 給定直方圖高度,求最大矩形面積。右邊界永遠是觸發清理的 `i`;左邊界則藏在 `popped` 自己裡——相鄰兩個被彈出者互為左邊界,最後一個的左邊界是 `prevTop`。

| 維度   | 具體問題                             | 抽象映射                                                            |
| ------ | ------------------------------------ | ------------------------------------------------------------------- |
| 維度 1 | 矩形受限於「更矮的柱子」→ 找「更小」 | 遞增棧,`shouldPop: heightAt(t) > heightAt(i)`                       |
| 維度 2 | 同時要左右邊界                       | 右邊界 = 觸發者 `i`;左邊界 = `popped` 的下一個,最後一個取 `prevTop` |
| 維度 3 | 面積 = 高 × 寬                       | 高 = 被彈出者的高;寬 = `i - 左邊界 - 1`                             |
| 維度 4 | 需哨兵收尾                           | 走到 `i = n` 時高度視為 0,觸發最後清理                              |

```javascript
function largestRectangleArea(heights) {
  const n = heights.length;
  const heightAt = (idx) => (idx === n ? 0 : heights[idx]); // 維度4：哨兵高度 0
  let maxArea = 0;
  // 維度1：遞增棧 —— 棧頂比新元素高就清理（被清理者的右邊界 = 當前 i）
  const ms = new MonotonicStack((i, t) => heightAt(t) > heightAt(i));
  for (let i = 0; i <= n; i++) {
    const { popped, prevTop } = ms.push(i); // 與題目 A 同一套批次介面
    // 維度2：popped 由棧頂往下排列 → 相鄰兩個互為左邊界，最後一個的左邊界是 prevTop
    for (let k = 0; k < popped.length; k++) {
      const left = k + 1 < popped.length ? popped[k + 1] : prevTop;
      const leftBound = left === undefined ? -1 : left; // 左邊無更矮者 → 視為 -1
      const width = i - leftBound - 1; // 維度3：右邊界(i) - 左邊界 - 1
      maxArea = Math.max(maxArea, heightAt(popped[k]) * width);
    }
  }
  return maxArea;
}
```

---

兩題現在都只做一件事:呼叫 `ms.push(i)`,然後從回傳的 `popped` / `prevTop` 把答案讀出來。差別純粹在「讀出來之後怎麼用」,不再是兩種介面。

## 🔍 觸發器（模式識別）

### 正向觸發器

| 層級         | 特徵                                                                     | 為什麼這個特徵指向 Monotonic Stack               |
| ------------ | ------------------------------------------------------------------------ | ------------------------------------------------ |
| **關鍵字**   | 「next greater」「next smaller」「previous greater」「previous smaller」 | 直接對應四種基本模式                             |
| **關鍵字**   | 「最近的」「第一個比它大／小的」                                         | 「最近」由棧的 LIFO 自然保證                     |
| **結構**     | 一維陣列 + 對每個元素找某種「鄰近大小關係」                              | Monotonic Stack 專處理一維陣列的近鄰查詢         |
| **結構**     | 直方圖、柱狀圖、天際線                                                   | 柱子的展開範圍受限於兩側更矮的柱子 → NSE         |
| **目標**     | 算每個元素的「有效範圍」（左右邊界）                                     | 有效範圍 = Previous Smaller 到 Next Smaller 之間 |
| **目標**     | 子陣列最值貢獻（每個元素作為最小／最大值的管轄範圍）                     | 該管轄範圍由 Monotonic Stack 決定                |
| **操作模式** | 暴力解的結構是「對每個元素線性掃描找特定條件」                           | 這正是 Monotonic Stack 要優化的模式              |

**一句話觸發規則**：當問題要對陣列中每個元素找「最近的更大／更小元素」，或算「受大小關係限制的有效範圍」時，考慮 Monotonic Stack。

### 反向觸發器

| 陷阱情境                                 | 為什麼不適用                                 | 更好的選擇                                   |
| ---------------------------------------- | -------------------------------------------- | -------------------------------------------- |
| 要「右邊所有更大者」或「依大小排名查詢」 | Monotonic Stack 只給最近的那一個             | **Segment Tree / BIT**（值域上的排名與計數） |
| 滑動視窗的最大／最小值                   | 視窗有左邊界，元素可能要從前端移除           | **Monotonic Deque**（雙端佇列）              |
| 二維矩陣最值（非可轉直方圖）             | 不是一維線性結構                             | **DP** 或 **Segment Tree 2D**                |
| 需要動態插入／刪除後仍維持查詢           | Monotonic Stack 是一次性遍歷，不支援動態更新 | **Balanced BST** 或 **Segment Tree**         |
| 找全域最大／最小值                       | 不需要「每個元素各自的」近鄰                 | 直接 `Math.max/min` 一次掃描                 |

### 與類似工具的決策點

```plaintext
「對每個元素找特定條件的鄰居」
├── 找「最近的更大／更小」→ Monotonic Stack
├── 找「滑動視窗內的最大／最小」→ Monotonic Deque
├── 找「任意區間內的最大／最小」→ Segment Tree / Sparse Table
└── 找「排序後的相對位置／排名」→ BST / BIT

「一維陣列的區間問題」
├── 每個元素的有效範圍由大小關係決定 → Monotonic Stack
├── 固定寬度滑動視窗 → Monotonic Deque / Sliding Window
├── 任意區間多次查詢 → Prefix Sum / Segment Tree
└── 連續子陣列求和／積 → Prefix Sum / Two Pointers

「直方圖／柱狀圖相關」
├── 最大矩形面積 → Monotonic Stack（遞增棧）
├── 接雨水（Trapping Rain Water）→ Monotonic Stack 或 Two Pointers
└── 天際線問題 → Monotonic Stack + 排序 或 Heap
```

## ⚠️ 注意事項

### 常見陷阱

| 陷阱                  | 錯誤寫法                                      | 正確寫法                                            | 為什麼                                                             |
| --------------------- | --------------------------------------------- | --------------------------------------------------- | ------------------------------------------------------------------ |
| **搞反單調方向**      | 找 NGE 卻用遞增棧的清理條件 `arr[i] < arr[t]` | 找 NGE 用遞減棧：`shouldPop: arr[i] > arr[t]`       | 找更大要清掉「比新元素小」的棧頂，新元素更大時才能觸發 pop         |
| **存值而非索引**      | `ms.push(arr[i])`                             | `ms.push(i)`                                        | 存索引才能回填 `result[idx]`，也能用 `arr[i]` 取值                 |
| **Previous 讀錯時機** | `ms.push(i)` 後才讀棧頂當答案                 | 讀 push 回傳的 `prevTop`（清理後、入棧前的棧頂）    | push 之後棧頂變成 `i` 自己，prevTop 已在 push 內部於正確時機取好   |
| **漏處理棧中殘留**    | 遍歷結束就回傳                                | 結束後仍在棧中者 = 沒有答案，確認 result 預設值正確 | 嚴格遞減輸入會讓所有元素都留在棧中                                 |
| **環形忘記取模**      | `arr[i]`                                      | `arr[i % n]`，並走 `2n` 圈                          | 環形要模擬「接在後面再走一圈」                                     |
| **等號處理不一致**    | 不分題意一律用 `>`                            | 依「相等是否算更大」決定 `>` 或 `>=`                | 等號直接決定相等元素是否並存（即棧為嚴格或非嚴格單調），影響正確性 |
| **哨兵遺漏**          | 直方圖題不加哨兵                              | 末尾加一根高度 0 的哨兵                             | 否則遍歷結束時遞增序列的柱子不會被清理計算                         |

### 常見變體

| 變體                         | 修改內容（改哪個操作的哪個部分）                                          | 適用場景                                         |
| ---------------------------- | ------------------------------------------------------------------------- | ------------------------------------------------ |
| **環形 Monotonic Stack**     | push 改用底層手動清理；走 `2n` 圈、取值用 `arr[i % n]`、只在 `i < n` 入棧 | LC 503 Next Greater Element II                   |
| **同時求左右邊界**           | 改用底層 pop/peek；彈出時右邊界 = `i`、左邊界 = 彈出後新棧頂              | LC 84 Largest Rectangle、LC 85 Maximal Rectangle |
| **貢獻法 + Monotonic Stack** | push 的 `popped` 用來算每個元素作為最小／最大值的管轄範圍 → 累加貢獻      | LC 907 Sum of Subarray Minimums、LC 2104         |
| **Previous Greater/Smaller** | 讀 push 回傳的 `prevTop` 而非 `popped`                                    | 需要「左邊最近的更大／更小」的問題               |
| **搭配 HashMap**             | 維持核心 push 不變，外層用 HashMap 做兩陣列的值→答案映射                  | LC 496 Next Greater Element I                    |

**環形版本參考實作**（屬變體，與核心 push 區隔；用底層操作）：

```javascript
function nextGreaterCircular(arr) {
  const n = arr.length;
  const result = new Array(n).fill(-1);
  const ms = new MonotonicStack();
  for (let i = 0; i < 2 * n; i++) {
    // 走 2n 圈模擬環形
    const val = arr[i % n];
    while (!ms.isEmpty() && val > arr[ms.peek()]) result[ms.pop()] = val;
    if (i < n) ms.push(i); // 只在第一圈入棧，第二圈僅負責清理
  }
  return result;
}
```
