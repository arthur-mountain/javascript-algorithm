# Stack (堆疊)

## 🎯 設計理念

### 存在意義

**這個工具被發明出來，是為了解決什麼類型的問題？**

- **在它出現之前的痛點**：當需要處理「最近的元素優先處理」的場景時（如函式呼叫、括號匹配、歷史記錄回溯），使用一般陣列需要手動追蹤「當前處理位置」，容易出錯且語義不清晰

- **核心價值主張**：強制執行 LIFO（Last-In-First-Out）訪問模式，確保只能從同一端進行插入與移除操作，自動管理「當前頂端」的位置

- **一句話本質**：專門管理「最近的元素優先處理」的受限陣列

```plaintext
痛點：用一般陣列處理「最後加入的先處理」場景時，需要手動維護索引且容易錯誤訪問中間元素
價值主張：以 O(1) 時間完成「插入最新元素」與「移除最新元素」
一句話本質：強制 LIFO 訪問順序的受限陣列
```

### 心智模型

**用生活化的類比建立直覺**

**類比：疊盤子**

- **Push**：把新洗好的盤子疊在最上面
- **Pop**：拿走最上面的盤子使用
- **Peek**：看一眼最上面是什麼盤子，但不拿走
- **只能從頂端操作**：你不會從中間或底部抽盤子（會倒塌）

**局限性**：

- 真實疊盤子可以「偷看」中間的盤子，但 Stack 不允許隨機訪問
- 疊盤子有物理高度限制，但 Stack 可以動態擴容（使用 Dynamic Array 實作時）

## 🏗️ 抽象結構

### 核心組件

| 組件      | 功能                     | 為什麼需要                                     |
| --------- | ------------------------ | ---------------------------------------------- |
| `data`    | 底層儲存結構（陣列）     | 實際儲存元素                                   |
| `top`     | 堆疊頂端索引             | 追蹤當前可以插入/移除的位置                    |
| `size`    | 當前元素數量（可選）     | 快速回傳堆疊大小，避免每次都要計算 `top + 1`   |
| `maxSize` | 最大容量（固定大小實作） | 防止無限增長（在記憶體受限環境）或簡化邊界檢查 |

### 狀態表示

- **使用什麼底層資料結構**：

  - **陣列實作**（常見）：使用 Dynamic Array，支援動態擴容
  - **鏈結串列實作**（較少見）：使用 Singly Linked List，每次在 head 插入/刪除

- **初始狀態**：

  ```javascript
  // 陣列實作
  data = [];
  top = -1; // 表示堆疊為空
  ```

- **關鍵狀態的定義規則**：
  - **空堆疊**：`top === -1` 或 `size === 0`
  - **滿堆疊**（固定大小）：`top === maxSize - 1` 或 `size === maxSize`
  - **有效索引範圍**：`[0, top]`

### 核心操作

#### 1. Push（插入）

**目的**：在堆疊頂端加入新元素

**輸入/輸出**：

- 輸入：要插入的值 `value`
- 輸出：無（修改內部狀態）

**不變量**：

- 執行後 `top` 指向新插入的元素
- 執行後堆疊大小增加 1
- 新元素成為「最新」的元素

**基礎實作**：

```javascript
push(value) {
  // 直覺做法：直接插入到 top + 1 位置
  this.top++;
  this.data[this.top] = value;
}
```

**基礎版本的問題**：

- **最壞情況**：當底層陣列容量不足時，需要擴容（若使用固定陣列則會 overflow）
- **複雜度退化**：
  - 固定陣列：達到 `maxSize` 時無法插入
  - Dynamic Array：擴容時需要 O(n) 複製所有元素

**優化策略**：使用 Dynamic Array 的擴容機制

**核心思想**：當容量不足時，分配更大的陣列（通常是 2 倍）並複製元素

**為什麼這樣能解決問題**：雖然單次擴容是 O(n)，但攤銷後每次 push 仍是 O(1)

```javascript
push(value) {
  // 檢查是否需要擴容
  if (this.top + 1 >= this.data.length) {
    this.resize(this.data.length * 2);
  }
  this.top++;
  this.data[this.top] = value;
}

resize(newCapacity) {
  const newData = new Array(newCapacity);
  for (let i = 0; i <= this.top; i++) {
    newData[i] = this.data[i];
  }
  this.data = newData;
}
```

**視覺化說明**：

```
初始 capacity = 2:  [A, B]  top = 1
                            ↑

Push(C) 觸發擴容:
舊陣列:            [A, B]
新陣列 (cap=4):    [A, B, _, _]
插入 C:            [A, B, C, _]  top = 2
                            ↑
```

**優化後的複雜度**：

- **時間**：O(1) 攤銷 / O(n) 最壞（擴容時）
- **空間**：O(1) 攤銷 / O(n) 最壞（擴容時需要新陣列）

#### 2. Pop（移除）

**目的**：移除並回傳堆疊頂端的元素

**輸入/輸出**：

- 輸入：無
- 輸出：堆疊頂端的值，若堆疊為空則拋出錯誤或回傳 `null`

**不變量**：

- 執行後 `top` 減 1
- 執行後堆疊大小減 1
- 移除的元素是「最新」加入的元素

**基礎實作**：

```javascript
pop() {
  // 邊界檢查
  if (this.isEmpty()) {
    throw new Error("Stack underflow");
  }
  // 取得頂端值並將 top 指標下移
  const value = this.data[this.top];
  this.top--;
  return value;
}
```

**基礎版本的問題**：

- **記憶體洩漏**：在 JavaScript 中，雖然 `top` 減少了，但舊元素仍然被 `data` 陣列引用，可能導致記憶體無法回收
- **空間浪費**：大量 pop 後，底層陣列容量仍然很大

**優化策略**：清空引用 + 動態縮容

**核心思想**：

1. Pop 時主動清空舊元素的引用
2. 當使用率過低時（如 `size < capacity / 4`），縮小陣列容量

**為什麼這樣能解決問題**：避免記憶體洩漏並維持合理的空間使用率

```javascript
pop() {
  if (this.isEmpty()) {
    throw new Error("Stack underflow");
  }

  const value = this.data[this.top];
  this.data[this.top] = null; // 清空引用，幫助 GC
  this.top--;

  // 動態縮容：當使用率低於 25% 時，縮小為一半
  if (this.top + 1 > 0 && this.top + 1 < this.data.length / 4) {
    this.resize(Math.floor(this.data.length / 2));
  }

  return value;
}
```

**視覺化說明**：

```
初始狀態 (cap=8):  [A, B, C, D, _, _, _, _]  top = 3
                               ↑

Pop D:             [A, B, C, null, _, _, _, _]  top = 2
                         ↑

Pop C:             [A, B, null, null, _, _, _, _]  top = 1
                      ↑

Pop B (觸發縮容):   [A, _, _, _]  top = 0  (cap 從 8 → 4)
                    ↑
```

**優化後的複雜度**：

- **時間**：O(1) 攤銷 / O(n) 最壞（縮容時）
- **空間**：O(1)

#### 3. Peek（查看頂端）

**目的**：查看但不移除堆疊頂端的元素

**輸入/輸出**：

- 輸入：無
- 輸出：堆疊頂端的值，若堆疊為空則拋出錯誤或回傳 `null`

**不變量**：

- 堆疊狀態完全不變
- `top` 指標不變

**基礎實作**：

```javascript
peek() {
  if (this.isEmpty()) {
    throw new Error("Stack is empty");
  }
  return this.data[this.top];
}
```

**基礎版本的問題**：無（已是最優實作）

**複雜度**：

- **時間**：O(1)
- **空間**：O(1)

---

#### 4. isEmpty（檢查是否為空）

**目的**：判斷堆疊是否沒有任何元素

**輸入/輸出**：

- 輸入：無
- 輸出：布林值

**基礎實作**：

```javascript
isEmpty() {
  return this.top === -1;
}
```

**複雜度**：

- **時間**：O(1)
- **空間**：O(1)

### 核心操作複雜度總結

| 操作           | 時間複雜度                | 空間複雜度            | 備註                             |
| -------------- | ------------------------- | --------------------- | -------------------------------- |
| **插入**       |                           |                       |                                  |
| push(val)      | **O(1) 攤銷** / O(n) 最壞 | O(1) 攤銷 / O(n) 最壞 | 滿時擴容為 capacity × 2          |
| **刪除**       |                           |                       |                                  |
| pop()          | **O(1) 攤銷** / O(n) 最壞 | O(1)                  | 使用率 < 25% 時縮為 capacity / 2 |
| **查詢**       |                           |                       |                                  |
| peek()         | **O(1)**                  | O(1)                  | 只讀取，不修改狀態               |
| isEmpty()      | **O(1)**                  | O(1)                  | 單純檢查 `top === -1`            |
| **內部操作**   |                           |                       |                                  |
| resize(newCap) | O(n)                      | O(n)                  | 分配新陣列並複製有效元素         |

**符號定義**：

- `n`：當前元素數量（`top + 1`）
- **攤銷 (Amortized)**：平均每次操作的時間成本
- **最壞 (Worst Case)**：單次操作可能的最大成本

**結構本身的空間複雜度**：O(capacity)，最壞情況 capacity ≈ 4 × n（動態縮容維持）

## ⭐ 抽象化翻譯器

### 識別核心抽象

> **這個工具的核心對象是**：元素與其插入順序
>
> **它管理的是這些對象之間的什麼關係**：強制 LIFO（後進先出）的訪問順序，只能從同一端（頂端）進行插入與移除

### 建立映射維度

| 維度       | 要回答的問題                         | 這個答案決定了什麼           |
| ---------- | ------------------------------------ | ---------------------------- |
| **維度 1** | 問題中的「什麼元素」需要 LIFO 訪問？ | Stack 儲存的資料類型         |
| **維度 2** | 什麼時候需要「記住」某個元素？       | 何時呼叫 `push()`            |
| **維度 3** | 什麼時候需要「使用最近記住的」元素？ | 何時呼叫 `pop()` 或 `peek()` |
| **維度 4** | 是否需要在使用後保留該元素？         | 使用 `peek()` 還是 `pop()`   |

### 映射範例

#### 範例 1：Valid Parentheses（有效的括號）

**題目**：判斷字串中的括號是否配對正確（`()`, `{}`, `[]`）

| 維度       | 具體問題         | 抽象映射                          |
| ---------- | ---------------- | --------------------------------- |
| **維度 1** | 儲存什麼？       | 左括號字元（`'('`, `'{'`, `'['`） |
| **維度 2** | 何時 Push？      | 遇到左括號時                      |
| **維度 3** | 何時 Pop/Peek？  | 遇到右括號時                      |
| **維度 4** | 使用後是否保留？ | 否，配對成功後移除（`pop()`）     |

**從映射到實作的關鍵步驟**：

1. 遍歷字串的每個字元
2. 若是左括號 → `push(字元)`
3. 若是右括號 → 檢查 `peek()` 是否為對應的左括號
   - 是 → `pop()` 並繼續
   - 否 → 回傳 `false`
4. 最後檢查 Stack 是否為空（所有括號都配對完成）

```javascript
function isValid(s) {
  const stack = [];
  const pairs = { ")": "(", "}": "{", "]": "[" };

  for (const char of s) {
    if (char === "(" || char === "{" || char === "[") {
      stack.push(char); // 維度 2：遇到左括號，Push
    } else {
      // 維度 3：遇到右括號，Pop/Peek
      if (stack.length === 0 || stack[stack.length - 1] !== pairs[char]) {
        return false;
      }
      stack.pop(); // 維度 4：配對成功，移除
    }
  }

  return stack.length === 0; // 所有括號都應該配對完成
}
```

#### 範例 2：Daily Temperatures（每日溫度）

**題目**：給定每日溫度陣列，回傳「需要等幾天才會有更高溫度」的陣列

| 維度       | 具體問題         | 抽象映射                            |
| ---------- | ---------------- | ----------------------------------- |
| **維度 1** | 儲存什麼？       | 尚未找到「更高溫度」的日期索引      |
| **維度 2** | 何時 Push？      | 每處理一個日期時                    |
| **維度 3** | 何時 Pop/Peek？  | 當前溫度比 Stack 頂端日期的溫度高時 |
| **維度 4** | 使用後是否保留？ | 否，找到答案後移除（`pop()`）       |

**從映射到實作的關鍵步驟**：

1. 建立結果陣列 `result`，初始化為 0
2. 遍歷溫度陣列（從左到右）
3. 對於當前日期 `i`：
   - 當 Stack 非空 且 `temperatures[i] > temperatures[stack.peek()]`：
     - 彈出索引 `prevIdx = stack.pop()`
     - 設定 `result[prevIdx] = i - prevIdx`（等待天數）
   - Push 當前索引 `i`

```javascript
function dailyTemperatures(temperatures) {
  const stack = []; // 儲存索引
  const result = new Array(temperatures.length).fill(0);

  for (let i = 0; i < temperatures.length; i++) {
    // 維度 3：當前溫度更高時，處理所有等待中的日期
    while (
      stack.length > 0 &&
      temperatures[i] > temperatures[stack[stack.length - 1]]
    ) {
      const prevIdx = stack.pop(); // 維度 4：找到答案，移除
      result[prevIdx] = i - prevIdx;
    }
    stack.push(i); // 維度 2：記住當前日期
  }

  return result;
}
```

## 🔍 觸發器（模式識別）

### 正向觸發器

看到什麼特徵應該想到 Stack？

| 層級         | 特徵                                        | 為什麼這個特徵指向 Stack                |
| ------------ | ------------------------------------------- | --------------------------------------- |
| **關鍵字**   | 「最近的」、「配對」、「巢狀」、「回溯」    | Stack 天生支援 LIFO，最近的元素最先處理 |
| **結構**     | 括號配對、函式呼叫、遞迴轉迭代              | 需要「記住」待處理的狀態，後處理先完成  |
| **目標**     | 找到「下一個更大/更小」、「等待某條件滿足」 | Monotonic Stack 模式                    |
| **操作模式** | 插入與移除發生在同一端                      | Stack 只允許單端操作，符合訪問限制      |

**一句話觸發規則**：當問題需要「記住最近的狀態並在滿足條件時回溯處理」時，考慮 Stack

### 反向觸發器

什麼情況下「看起來適用但其實不是」？

| 陷阱情境                   | 為什麼不適用                         | 更好的選擇                    |
| -------------------------- | ------------------------------------ | ----------------------------- |
| 需要查找中間元素           | Stack 不支援隨機訪問，只能從頂端操作 | Array 或 Deque                |
| 需要同時從兩端操作         | Stack 只支援單端操作                 | Deque                         |
| 需要找最小值但元素無序插入 | 普通 Stack 無法快速找最小值          | Min Heap 或 Min Stack（變體） |
| 需要 FIFO（先進先出）      | Stack 是 LIFO，不符合 FIFO 語義      | Queue                         |

### 與類似工具的決策點

```plaintext
需要「記住」元素並稍後處理
├── 處理順序是「最新的優先」？ → Stack
├── 處理順序是「最早的優先」？ → Queue
├── 需要同時從兩端操作？ → Deque
└── 需要按優先級處理？ → Heap (Priority Queue)
```

## ⚠️ 注意事項

### 常見陷阱

| 陷阱                     | 錯誤寫法                                 | 正確寫法                                 | 為什麼                             |
| ------------------------ | ---------------------------------------- | ---------------------------------------- | ---------------------------------- |
| 空堆疊時 Pop             | `stack.pop()`（未檢查）                  | `if (!stack.isEmpty()) stack.pop()`      | 避免 underflow 錯誤                |
| 使用陣列長度判斷是否為空 | `if (stack.length === 0)`                | `if (stack.isEmpty())`                   | 封裝邊界邏輯，避免直接訪問內部狀態 |
| 混淆 Peek 與 Pop         | `let val = stack.pop(); stack.push(val)` | `let val = stack.peek()`                 | Peek 不改變狀態，Pop 會移除元素    |
| 忘記清空引用（JS/Java）  | `this.top--`（未設 `null`）              | `this.data[this.top] = null; this.top--` | 幫助 GC 回收記憶體，避免記憶體洩漏 |

### 常見變體

| 變體                | 修改內容                       | 適用場景                               |
| ------------------- | ------------------------------ | -------------------------------------- |
| **Min Stack**       | 額外維護一個「最小值 Stack」   | 需要 O(1) 時間取得當前最小值           |
| **Monotonic Stack** | 維持 Stack 內元素單調遞增/遞減 | 找「下一個更大/更小元素」              |
| **固定大小 Stack**  | 限制最大容量，不擴容           | 記憶體受限環境、有明確大小上限         |
| **鏈結串列實作**    | 使用 Linked List 代替陣列      | 不需要擴容、頻繁 push/pop 且不關心容量 |
